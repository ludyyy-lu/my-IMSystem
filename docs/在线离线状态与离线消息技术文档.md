# 在线/离线状态管理与离线消息缓存 — 技术文档

## 一、概述

本文档描述 `ws-gateway` 服务中**用户在线/离线状态管理**与**离线消息缓存推送**两个核心功能的设计、实现细节及运维要点。

---

## 二、核心组件

```
ws-gateway
└── internal/
    ├── session/
    │   ├── session.go          # 单条 WebSocket 连接生命周期管理
    │   ├── manager.go          # 在线 Session 路由表（内存级）
    │   ├── state.go            # State 类型定义
    │   ├── presence.go         # Redis 跨节点在线状态存储（PresenceStore）
    │   └── offline_store.go    # Redis 离线消息缓存（OfflineStore）
    ├── push/
    │   └── service.go          # 消息推送服务（在线直推 + 离线缓存）
    └── transport/
        └── handler.go          # HTTP→WS 升级、状态标记、离线消息投递
```

## 三、在线/离线状态管理

### 3.1 两层状态存储

系统使用**两层互补**的状态存储：

| 存储层 | 类型 | 作用域 | 读写延迟 |
|--------|------|--------|---------|
| `session.Manager` | 进程内 `map[int64]*Session` | 单节点 | 纳秒级 |
| `session.RedisPresenceStore` | Redis `SET` 键值 | 所有节点共享 | 毫秒级 |

**为何需要两层？**

- 单节点内部消息路由（`push.Service.PushToUser`）直接查内存 Map，零延迟。
- 多节点部署时，下游服务（如 `friend-service`、HTTP 接口）需要知道用户是否在线，必须通过 Redis 查询跨节点状态。
- 网关节点崩溃时，Redis 键会在 TTL 到期后自动失效，避免"僵尸在线"状态。

### 3.2 PresenceStore 接口

```go
// ws-gateway/internal/session/presence.go

type PresenceStore interface {
    SetOnline(ctx context.Context, userID int64) error
    SetOffline(ctx context.Context, userID int64) error
    IsOnline(ctx context.Context, userID int64) (bool, error)
}
```

Redis 键格式：`presence:online:{userID}`，值为 `1`，TTL 为 **2 小时**。

TTL 的作用：
- 正常断开时 `SetOffline` 立即删除键。
- 异常崩溃（网关进程被 kill -9）时，键在 2 小时后自动过期，不会留下永久的"在线"幻象。

### 3.3 状态流转

```
用户发起 WS 连接
       │
       ▼
ConnectHandler.VerifyToken() ── 失败 ──► HTTP 401, 结束
       │ 成功
       ▼
upgrader.Upgrade()  (HTTP 101 Switching Protocols)
       │
       ▼
PresenceStore.SetOnline(ctx, userID)      ← 写 Redis
       │
       ▼
SessionManager.Add(sess)                  ← 写内存 Map
       │
       ▼
sess.Start()  (启动 readLoop + writeLoop goroutine)
       │
       ▼
go deliverOfflineMessages(...)            ← 异步投递离线消息
       │
       ▼
[连接活跃中 ...]
       │
       ▼ （客户端断开 / 读超时 / 服务端主动关闭）
sess.Close()
       │
       ├─► SessionManager.Remove(uid)     ← 清理内存 Map
       └─► PresenceStore.SetOffline(...)  ← 删除 Redis 键
```

### 3.4 Manager.IsOnline — 本地快速查询

```go
// manager.go
func (m *Manager) IsOnline(userID int64) bool {
    _, ok := m.Get(userID)
    return ok
}
```

适用于**同一网关节点**内的快速在线判断（无 Redis 访问）。  
跨节点场景请使用 `PresenceStore.IsOnline`。

## 四、离线消息缓存与上线推送

### 4.1 整体流程

```
Kafka 消费到一条消息
       │
       ▼
push.Service.PushToUser(userID, msgType, payload)
       │
       ├─ [用户在线] ──► SessionManager.SendTo(userID, data)
       │                        │
       │                        ▼
       │                  session.Send → writeLoop → WS帧 → 客户端
       │
       └─ [用户离线] ──► OfflineStore.Save(userID, data)
                                │
                                ▼
                          Redis RPUSH offline:msg:{userID}
                          （存入预序列化的 PushMessage JSON）
```

用户重新上线时：

```
ConnectHandler 建立连接 & 注册 Session
       │
       ▼
go deliverOfflineMessages(userID, svcCtx, sess)
       │
       ▼
OfflineStore.LoadAndDelete(userID)
  → Redis LRANGE offline:msg:{userID} 0 -1
  → Redis DEL   offline:msg:{userID}     （原子清除）
       │
       ▼
for each data:
    sess.Send(data)    （直接发送，无需再次序列化）
       │
       ▼
客户端收到原始类型的消息（chat_message / friend_event / ...）
```

### 4.2 OfflineStore 接口

```go
// ws-gateway/internal/session/offline_store.go

type OfflineStore interface {
    // Save 将预序列化的 PushMessage 字节追加到用户的离线队列。
    Save(userID int64, data []byte) error

    // LoadAndDelete 读取并原子清除用户的所有离线消息。
    LoadAndDelete(userID int64) ([][]byte, error)
}
```

**设计要点：存储原始字节，而非结构体。**

早期设计将 `model.WsMessage` 存入离线队列，导致两个问题：

1. 离线队列只能缓存聊天消息，无法处理好友通知（`FriendEvent`）等其他类型。
2. 上线投递时需要再次序列化，引入不必要的 CPU 消耗。

新设计在 `push.Service` 层统一序列化为 `PushMessage` JSON 后再存储，  
上线时直接将字节切片送入 WebSocket 写队列，零拷贝投递。

### 4.3 Redis 键结构

| 键名 | 数据类型 | 说明 |
|------|---------|------|
| `offline:msg:{userID}` | Redis List | 离线消息队列，元素为 JSON 字节 |
| `presence:online:{userID}` | Redis String | 在线标记，值固定为 `1` |

### 4.4 push.Service 核心逻辑

```go
// ws-gateway/internal/push/service.go

func (s *Service) PushToUser(userID int64, messageType string, payload interface{}) {
    data, _ := json.Marshal(PushMessage{Type: messageType, Payload: payload})

    if err := s.manager.SendTo(userID, data); err != nil {
        // 用户离线：将预序列化字节存入 OfflineStore
        if s.offlineStore != nil {
            s.offlineStore.Save(userID, data)
        }
    }
}
```

`push.Service` 现在接受可选的 `session.OfflineStore`。  
当 `offlineStore == nil` 时退化为原始行为（仅记录日志），保持向后兼容。

## 五、多节点部署注意事项

### 5.1 单节点 vs. 多节点的状态同步

| 场景 | Manager.IsOnline | PresenceStore.IsOnline |
|------|-----------------|----------------------|
| 单节点 | ✅ 可用 | ✅ 可用 |
| 多节点（用户连在同一节点） | ✅ | ✅ |
| 多节点（用户连在其他节点） | ❌ 返回 false | ✅ 返回 true |

**多节点部署时，业务层应使用 `PresenceStore.IsOnline` 判断全局在线状态。**

### 5.2 Kafka 消费者 + 离线缓存

```
用户 A (Node-1)          用户 B (Node-2)          Kafka
     │                        │                      │
     │ 发送消息 ──────────────► topic: chat_msgs ──► │
     │                        │                      │
     │                      Node-1 消费──► PushToUser(B)
     │                        │
     │                  B 连在 Node-2, Node-1 内存中没有 B 的 session
     │                        │
     │                  OfflineStore.Save(B, data) ── Redis
     │                        │
     │                  [B 重连 Node-1 或 Node-2]
     │                        │
     │                  deliverOfflineMessages ──► 从 Redis 读取并推送
```

> **注意**：当前实现中，Kafka 消费者总是将消息投递给本节点的 `push.Service`，  
> 若用户连接在其他节点，消息将被存入 Redis 离线队列。  
> 下次用户重连时（无论连到哪个节点），离线消息都会被取出并推送。

## 六、错误处理与容错

| 场景 | 处理策略 |
|------|---------|
| Redis 不可用，SetOnline 失败 | 记录 ERROR 日志，连接继续建立（降级：内存路由仍可用） |
| Redis 不可用，SetOffline 失败 | 记录 ERROR 日志，session 已从内存中移除，Redis 键等待 TTL 自然过期 |
| Redis 不可用，OfflineStore.Save 失败 | 记录 ERROR 日志，消息丢失（可接受的降级行为） |
| Redis 不可用，LoadAndDelete 失败 | 记录 ERROR 日志，离线消息暂时不投递，下次连接时重试 |
| PresenceStore 为 nil（测试/单测） | `if svcCtx.PresenceStore != nil` 守卫，安全跳过 |
| OfflineStore 为 nil | `if s.offlineStore != nil` 守卫，退化为日志记录 |

## 七、性能考量

### 7.1 离线消息投递是异步的

```go
// handler.go
sess.Start()
go deliverOfflineMessages(userID, svcCtx, sess)  // ← goroutine，不阻塞连接建立
```

连接建立后立即返回，离线消息在后台推送，不影响连接握手延迟。

### 7.2 批量推送 vs. 逐条推送

当前实现逐条调用 `sess.Send(data)`，每条消息进入容量为 256 的带缓冲 channel。  
如果离线消息量超过 256 条，`Send` 将返回 "send channel full" 错误并丢弃后续消息。

**生产环境建议：**
- 在 `RedisOfflineMsgStore.Save` 中使用 `LTRIM` 限制队列长度（如最多 200 条）：
  ```go
  s.rdb.RPush(ctx, key, data)
  s.rdb.LTrim(ctx, key, -200, -1)  // 保留最新 200 条
  ```
- 或在 `deliverOfflineMessages` 中分批推送，每批之间 `time.Sleep` 让出调度。

### 7.3 Redis TTL 防止内存泄漏

`presence:online:{userID}` 键 TTL = 2 小时，防止网关异常崩溃留下僵尸键。  
`offline:msg:{userID}` 列表当前**无 TTL**，极端情况下可能无限增长。  
生产环境建议设置 TTL（如 7 天）：
```go
s.rdb.Expire(ctx, key, 7*24*time.Hour)
```

## 八、接口契约（客户端协议）

### 8.1 上线推送（离线消息）

用户重连后，服务端通过 WebSocket 发送所有缓存消息，格式与实时消息**完全相同**：

```json
{"type": "chat_message", "payload": {...}}
{"type": "friend_event",  "payload": {...}}
```

客户端无需区分"实时消息"与"离线消息"，统一按 `type` 字段处理即可。

### 8.2 在线状态查询（未来扩展）

若需要向客户端暴露好友在线状态，可新增 WebSocket 推送类型：

```json
{"type": "presence_change", "payload": {"user_id": 42, "status": "online"}}
```

对应 `PresenceStore.IsOnline` 查询结果，通过 `push.Service.PushToUser` 推送给关注该用户的好友。
