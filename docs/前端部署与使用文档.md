## 四、本地开发环境

### 4.1 克隆仓库

```bash
git clone https://github.com/ludyyy-lu/my-IMSystem.git
cd my-IMSystem/frontend
```

### 4.2 安装依赖

```bash
npm install
```

### 4.3 启动后端基础设施

在启动前端之前，请确保后端服务已运行。最简单的方式是用 Docker Compose 启动基础设施：

```bash
# 在项目根目录执行
cd ../deploy
docker-compose up -d mysql redis etcd zookeeper kafka
```

然后手动启动各微服务（在各服务目录下）：

```bash
# auth-service（端口 8080）
cd auth-service && go run auth.go

# user-service（端口 8083）
cd user-service && go run user.go

# friend-service（端口 8082）
cd friend-service && go run friend.go

# chat-service（端口 8081）
cd chat-service && go run chat.go

# ws-gateway（端口 8888）
cd ws-gateway && go run ws.go

# api-gateway（端口 8090）
cd api-gateway && go run api.go
```

### 4.4 启动开发服务器

```bash
cd frontend
npm run dev
```

成功启动后终端会显示：

```
  VITE v8.x  ready in xxx ms

  ➜  Local:   http://localhost:3000/
  ➜  Network: use --host to expose
```

在浏览器中访问 **http://localhost:3000** 即可看到登录/注册页面。

> **代理机制说明**
>
> 开发模式下，Vite 内置代理会自动将以下请求转发：
> - `/api/*` → `http://localhost:8090`（api-gateway）
> - `/ws/*` → `ws://localhost:8888`（ws-gateway）
>
> 因此前端无需任何跨域配置，直接开发调试即可。

### 4.5 可用 npm 脚本

| 命令 | 说明 |
|------|------|
| `npm run dev` | 启动开发服务器（含热重载，端口 3000） |
| `npm run build` | 打包生产产物至 `dist/` 目录 |
| `npm run preview` | 本地预览生产构建效果（端口 4173） |
| `npm run lint` | 使用 ESLint 检查代码规范 |

## 五、环境变量配置

前端通过 Vite 的环境变量机制（`import.meta.env`）控制后端地址。

### 5.1 变量说明

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `VITE_API_URL` | `''`（空，使用 Vite 代理） | HTTP API 网关地址。开发模式留空，生产环境填写 api-gateway 的公网地址。 |
| `VITE_WS_URL` | 自动从 `window.location` 推导 | WebSocket 网关地址。开发模式留空，生产环境填写 ws-gateway 的 ws:// 地址。 |

### 5.2 开发模式（留空，走 Vite 代理）

无需创建 `.env` 文件，代理默认生效。若想覆盖，可在 `frontend/` 目录创建 `.env.local`：

```env
# frontend/.env.local（仅开发，不提交到 git）
VITE_API_URL=
VITE_WS_URL=
```

### 5.3 生产/自定义环境

创建 `frontend/.env.production`（会在 `npm run build` 时自动加载）：

```env
# 示例：后端部署在同一台服务器的 80/8090/8888 端口
VITE_API_URL=http://your-server-ip:8090
VITE_WS_URL=ws://your-server-ip:8888
```

如果前端和后端都在同一台机器上，且 nginx 统一对外暴露 80 端口并做反向代理，可以使用相对路径（留空），由 nginx 完成代理。

### 5.4 Docker 构建时传入环境变量

```bash
docker build \
  --build-arg VITE_API_URL=http://your-server:8090 \
  --build-arg VITE_WS_URL=ws://your-server:8888 \
  -t im-frontend .
```

## 六、生产环境构建

### 6.1 本地构建

```bash
cd frontend
npm run build
```

构建产物在 `dist/` 目录，可直接用任意静态服务器（nginx、caddy 等）托管。

```
dist/
├── index.html
└── assets/
    ├── index-xxxx.js
    └── index-xxxx.css
```

### 6.2 本地预览构建效果

```bash
npm run preview
# 访问 http://localhost:4173
```

### 6.3 使用 nginx 托管（手动）

将 `dist/` 内容复制到 nginx 的 `html` 目录，并配置类似 `frontend/nginx.conf` 的虚拟主机文件：

```nginx
server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # SPA：所有非静态资源路由均返回 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 代理 REST API 到 api-gateway
    location /api/ {
        proxy_pass http://api-gateway:8090;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # 代理 WebSocket 到 ws-gateway
    location /ws/ {
        proxy_pass http://ws-gateway:8888;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
}
```

> **重要**：`location /` 中的 `try_files $uri $uri/ /index.html` 是 SPA 单页应用的必要配置。缺少它时，直接访问 `/` 以外的路径会返回 404。

## 七、Docker 容器化部署

前端提供了完整的 `Dockerfile`，采用多阶段构建，最终镜像基于 `nginx:alpine`，体积极小。

### 7.1 构建镜像

```bash
# 在 frontend/ 目录执行
cd frontend

docker build -t im-frontend:latest .

# 也可以传入构建参数覆盖后端地址
docker build \
  --build-arg VITE_API_URL=http://192.168.1.100:8090 \
  --build-arg VITE_WS_URL=ws://192.168.1.100:8888 \
  -t im-frontend:latest .
```

### 7.2 运行容器

```bash
docker run -d \
  --name im-frontend \
  -p 3000:80 \
  im-frontend:latest
```

访问 http://localhost:3000 即可打开应用。

### 7.3 查看容器日志

```bash
docker logs -f im-frontend
```

### 7.4 停止/删除容器

```bash
docker stop im-frontend
docker rm im-frontend
```

### 7.5 Dockerfile 说明

```dockerfile
# 阶段一：Node.js 构建
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                        # 精确安装依赖（使用 package-lock.json）
COPY . .
ARG VITE_API_URL=http://api-gateway:8090   # 默认指向同 compose 网络中的服务名
ARG VITE_WS_URL=ws://ws-gateway:8888
ENV VITE_API_URL=${VITE_API_URL}
ENV VITE_WS_URL=${VITE_WS_URL}
RUN npm run build                 # 输出到 /app/dist

# 阶段二：nginx 服务静态文件
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

## 八、Docker Compose 一键部署（推荐）

使用项目根目录的 `deploy/docker-compose.yaml` 可以同时启动所有基础设施和应用服务。

### 8.1 启动全部服务

```bash
cd deploy
docker-compose up -d --build
```

等待所有容器启动（约 1~2 分钟）后：

```bash
docker-compose ps
```

正常状态应如下：

```
NAME               STATUS        PORTS
im-mysql           Up 2 min      0.0.0.0:3307->3306/tcp
im-redis           Up 2 min      0.0.0.0:6379->6379/tcp
im-etcd            Up 2 min      0.0.0.0:2379->2379/tcp
im-zookeeper       Up 2 min      0.0.0.0:2181->2181/tcp
im-kafka           Up 2 min      0.0.0.0:9092->9092/tcp, 0.0.0.0:9094->9094/tcp
im-api-gateway     Up 1 min      0.0.0.0:8090->8090/tcp
im-frontend        Up 1 min      0.0.0.0:3000->80/tcp
```

访问 **http://localhost:3000** 即可使用完整的 IM 系统。

### 8.2 仅启动前端相关服务

如果后端服务已单独部署，可只启动基础设施和前端：

```bash
docker-compose up -d mysql redis etcd zookeeper kafka api-gateway frontend
```

### 8.3 查看日志

```bash
# 查看前端容器日志
docker-compose logs -f frontend

# 查看 api-gateway 日志
docker-compose logs -f api-gateway
```

### 8.4 重新构建前端

修改前端代码后，重新构建并更新容器：

```bash
docker-compose up -d --build frontend
```

### 8.5 停止所有服务

```bash
docker-compose down
```

## 九、功能使用说明

### 9.1 注册账号

1. 打开 **http://localhost:3000**，默认显示登录页。
2. 点击底部 **「立即注册」**，切换到注册表单。
3. 填写**用户名**（须唯一）和**密码**，点击 **「注册」**。
4. 注册成功后自动跳转到聊天主界面，JWT Token 保存在浏览器 `localStorage`。

> **注意**：用户名一旦注册不可更改，请谨慎选择。

### 9.2 登录账号

1. 在登录页填写**用户名**和**密码**，点击 **「登录」**。
2. 登录成功后自动跳转到聊天主界面，WebSocket 连接自动建立。
3. 若 Token 仍有效（未手动清除 `localStorage`），刷新页面后会直接进入主界面，无需重新登录。

### 9.3 搜索用户并添加好友

1. 点击左侧边栏顶部的 **「搜索」** 标签页。
2. 在搜索框中输入对方**用户名**（支持模糊搜索，输入停止 400ms 后自动触发）。
3. 搜索结果列表中，点击目标用户右侧的 **「添加」** 按钮。
4. 系统弹出提示"好友申请已发送！"，等待对方确认。

### 9.4 处理好友申请

1. 点击左侧边栏的 **「申请」** 标签页，自动加载所有待处理的好友申请。
2. 每条申请显示申请人用户名和备注信息。
3. 点击 **「接受」** 同意好友申请；点击 **「拒绝」** 拒绝申请。
4. 接受后，该好友会立即出现在 **「好友」** 列表中。

### 9.5 发送消息（实时聊天）

1. 点击 **「好友」** 标签页，在好友列表中点击目标好友。
2. 右侧聊天窗口会加载最近 50 条历史消息，并自动滚动到最新一条。
3. 在底部输入框中输入消息内容：
   - 按 **Enter** 键发送消息。
   - 按 **Shift + Enter** 换行。
   - 也可以点击右侧 **「发送」** 按钮。
4. 消息发送后会立即显示（乐观更新），同时通过 WebSocket 实时推送给对方。

> **WebSocket 状态指示器**：聊天窗口右上角显示连接状态。
> - **● 已连接**（绿色）：WebSocket 正常，消息可实时收发。
> - **○ 断开**（红色）：连接断开，系统将每 3 秒自动尝试重连，期间无法发送新消息（「发送」按钮变灰）。

### 9.6 未读消息提醒

- 收到新消息时，若对应好友未被选中（未打开聊天窗口），该好友右侧会显示红色**未读消息角标**（显示未读数量）。
- 点击对应好友打开聊天窗口后，未读角标自动清零，并向服务端发送**批量已读回执**（`batch_ack`）。

### 9.7 退出登录

点击左侧边栏右上角的 **「退出」** 按钮，清除 `localStorage` 中的 Token 并返回登录页。

## 十、常见问题排查

### 10.1 启动后访问页面空白或报错 404

**原因**：nginx 缺少 SPA 路由配置，或构建产物未正确复制。

**解决**：
- 确认 `nginx.conf` 中 `location /` 包含 `try_files $uri $uri/ /index.html`。
- Docker 部署时，确认镜像构建成功（`docker build` 无报错）。

### 10.2 点击登录/注册后提示"Request failed"或网络错误

**原因**：前端无法连接到 api-gateway。

**解决步骤**：
1. 确认 api-gateway 正在运行：
   ```bash
   curl http://localhost:8090/api/auth/login
   # 应返回 JSON，而非 "connection refused"
   ```
2. 确认 Vite 代理配置正确（`vite.config.js` 中 `/api` 代理目标为 `http://localhost:8090`）。
3. Docker Compose 部署时，确认 `im-api-gateway` 容器状态为 `Up`：
   ```bash
   docker-compose ps api-gateway
   ```

### 10.3 能登录但 WebSocket 无法连接（显示"○ 断开"）

**原因**：ws-gateway 未运行，或 WebSocket 代理配置有误。

**解决步骤**：
1. 确认 ws-gateway 正在运行（端口 8888）：
   ```bash
   # 查看端口占用
   lsof -i :8888
   ```
2. 开发模式下，确认 `vite.config.js` 中 `/ws` 代理已配置：
   ```js
   '/ws': {
     target: 'ws://localhost:8888',
     ws: true,
     changeOrigin: true,
   }
   ```
3. 浏览器控制台（F12 → Console / Network → WS）查看 WebSocket 握手是否成功，检查具体错误信息。

### 10.4 搜索用户无结果

**原因**：user-service 未运行，或 Etcd 服务发现失败。

**解决**：
- 确认 user-service 已注册到 Etcd：
  ```bash
  etcdctl get "" --prefix --keys-only | grep user
  ```
- 查看 api-gateway 日志确认 gRPC 调用是否报错：
  ```bash
  docker-compose logs -f api-gateway
  ```

### 10.5 发送消息后对方未收到

**原因**：Kafka 消息未正常消费，或对方未在线（WebSocket 未连接）。

**解决**：
1. 确认 Kafka 正常运行：
   ```bash
   docker-compose logs kafka | tail -20
   ```
2. 确认 chat-service 的 Kafka consumer 正在消费 `im-chat-topic`。
3. 若对方不在线，消息会存入离线消息队列，下次上线后自动下发。

### 10.6 页面刷新后需要重新登录

**原因**：`localStorage` 中的 Token 被清除，或浏览器的隐私模式。

**说明**：这是正常安全行为。若 Token 仍在 `localStorage`，刷新不会强制重新登录。

### 10.7 Docker 构建失败：node_modules 问题

```bash
# 清理 node_modules 后重新安装
cd frontend
rm -rf node_modules package-lock.json
npm install
```

### 10.8 修改 `VITE_API_URL` 后不生效

Vite 环境变量在构建时**静态注入**（`import.meta.env` 在构建时被替换为字面量），修改后必须重新构建：

```bash
npm run build
# 或 Docker
docker-compose up -d --build frontend
```
