# IM即时通讯系统 - 技术亮点与实现细节

## 一、核心技术亮点

### 1.1 微服务架构设计

#### 架构优势

本项目采用**基于 go-zero 的微服务架构**，将系统拆分为多个独立服务，每个服务职责单一、边界清晰：

```
auth-service    → 专注认证授权
user-service    → 专注用户管理
chat-service    → 专注消息处理
friend-service  → 专注好友关系
ws-gateway      → 专注长连接管理
```

**技术优势：**
1. **独立部署**：每个服务可以独立升级，不影响其他服务
2. **技术异构**：不同服务可以选择最适合的技术栈（虽然本项目统一用 Go）
3. **故障隔离**：单个服务故障不会导致整个系统崩溃
4. **水平扩展**：可以根据负载单独扩展某个服务（如 chat-service 高负载时多部署几个实例）

#### 服务间通信机制

**gRPC + Protobuf**
```protobuf
// common/proto/auth/auth.proto
service Auth {
    rpc Login(LoginRequest) returns (LoginResponse);
    rpc VerifyToken(VerifyTokenRequest) returns (VerifyTokenResponse);
    rpc RefreshToken(RefreshTokenRequest) returns (RefreshTokenResponse);
}

message LoginRequest {
    string username = 1;
    string password = 2;
    string device_id = 3;
}
```

**为什么选择 gRPC？**
- **性能**：二进制序列化，比 JSON 快 3-5 倍
- **强类型**：Protobuf 定义接口契约，编译期发现错误
- **多语言支持**：未来前端可以用 Node.js、Python 等调用
- **流式传输**：支持 Server Streaming（如实时推送通知）

**实际代码示例：**
```go
// ws-gateway 调用 auth-service 验证 Token
func (h *Handler) verifyToken(token string) (int64, error) {
    resp, err := h.authClient.VerifyToken(context.Background(), &auth.VerifyTokenRequest{
        AccessToken: token,
    })
    if err != nil {
        return 0, err
    }
    return resp.UserId, nil
}
```

### 1.2 Kafka 异步消息架构

#### 为什么选择 Kafka？

在 IM 系统中，消息发送和接收是**解耦**的：
- 发送者只需要将消息投递到 Kafka
- 接收者通过 WebSocket 连接从 Kafka 消费消息

**Kafka 三大核心优势：**

| 特性 | 解决的问题 |
|------|-----------|
| **削峰填谷** | 瞬时高并发消息不会直接打垮 WebSocket 服务 |
| **持久化** | 消息写入磁盘，即使消费者宕机也不会丢失 |
| **水平扩展** | 通过分区机制支持百万级消息吞吐 |

#### 消息分区策略

**核心代码：**
```go
// chat-service/internal/logic/send_message_logic.go
err := l.svcCtx.KafkaProducer.WriteMessages(context.Background(),
    kafka.Message{
        Topic: "im-chat-topic",
        Key:   []byte(strconv.FormatInt(req.ToUserId, 10)), // 以接收者为分区键
        Value: msgBytes,
    },
)
```

**为什么以 `ToUserId` 作为分区键？**
1. **消息顺序性**：同一用户的所有消息路由到同一分区，保证严格有序
2. **负载均衡**：不同用户的消息分布在不同分区，实现自然负载均衡
3. **高效消费**：每个 WebSocket 网关实例消费固定分区，避免锁竞争

#### 消息流转全链路

```
┌──────────────┐
│ 用户 A 发消息  │
└──────┬───────┘
       │
       ▼
┌─────────────────────┐
│ WS Gateway 接收      │
└──────┬──────────────┘
       │ gRPC 调用
       ▼
┌─────────────────────┐
│ Chat Service        │
│ 1. 检查黑名单        │
│ 2. 生成消息 ID       │
│ 3. 写入 Kafka        │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ Kafka Topic         │
│ Partition 3         │  ← 根据 ToUserId 分区
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ WS Gateway Consumer │
│ 消费分区 3 的消息     │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ Connection Manager  │
│ 查找用户 B 的连接     │
└──────┬──────────────┘
       │
       ▼
┌─────────────────────┐
│ 用户 B 收到消息       │
└─────────────────────┘
```

**关键代码 - Kafka 消费者：**
```go
// ws-gateway/internal/kafka/consumer.go
func StartChatConsumer(manager *ws.Manager) {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers:  []string{"localhost:9092"},
        Topic:    "im-chat-topic",
        GroupID:  "ws-gateway-group",
        MinBytes: 10e3, // 10KB
        MaxBytes: 10e6, // 10MB
    })
    
    for {
        msg, err := reader.ReadMessage(context.Background())
        if err != nil {
            log.Printf("读取 Kafka 消息失败: %v", err)
            continue
        }
        
        var chatMsg ChatMessage
        json.Unmarshal(msg.Value, &chatMsg)
        
        // 推送给在线用户
        manager.PushToUser(chatMsg.ToUserId, msg.Value)
    }
}
```

### 1.3 WebSocket 长连接管理

#### 连接池设计

**核心数据结构：**
```go
// ws-gateway/internal/ws/manager.go
type Manager struct {
    Connections sync.Map  // map[int64]*Conn，线程安全的连接池
    mu          sync.RWMutex
}

type Conn struct {
    UserID   int64
    WsConn   *websocket.Conn
    SendCh   chan []byte        // 发送缓冲区（容量 256）
    CloseCh  chan struct{}      // 关闭信号
    IsClosed atomic.Bool        // 原子布尔，避免重复关闭
}
```

**为什么使用 `sync.Map` 而不是 `map + RWMutex`？**
- Go 1.9+ 提供的并发安全 Map
- 读多写少场景性能优于传统锁
- IM 场景正是：频繁读取连接，偶尔添加/删除

#### 读写分离架构

每个 WebSocket 连接有**两个独立的 goroutine**：

```go
// ws-gateway/internal/ws/conn.go
func (c *Conn) readLoop() {
    defer c.Close()
    for {
        _, message, err := c.WsConn.ReadMessage()
        if err != nil {
            return  // 连接断开，退出循环
        }
        // TODO: 处理客户端发来的消息
        // 例如：心跳响应、消息回执（ACK）等
    }
}

func (c *Conn) writeLoop() {
    ticker := time.NewTicker(30 * time.Second)
    defer ticker.Stop()
    
    for {
        select {
        case message := <-c.SendCh:
            // 发送消息给客户端
            c.WsConn.WriteMessage(websocket.TextMessage, message)
            
        case <-ticker.C:
            // 发送心跳 PING
            c.WsConn.WriteMessage(websocket.PingMessage, nil)
            
        case <-c.CloseCh:
            return  // 关闭信号，退出循环
        }
    }
}
```

**设计亮点：**
1. **读写分离**：避免阻塞，提高并发性能
2. **心跳保活**：30 秒发送一次 PING，检测僵尸连接
3. **缓冲队列**：256 容量的 SendCh，应对突发流量
4. **优雅关闭**：通过 CloseCh 通知 goroutine 退出，避免泄漏

#### 连接生命周期管理

```go
// 1. 建立连接
func (h *Handler) HandleWS(w http.ResponseWriter, r *http.Request) {
    // 验证 Token
    token := r.Header.Get("Authorization")
    userId, err := h.verifyToken(token)
    
    // 升级为 WebSocket
    wsConn, _ := upgrader.Upgrade(w, r, nil)
    
    // 创建连接对象
    conn := &Conn{
        UserID:  userId,
        WsConn:  wsConn,
        SendCh:  make(chan []byte, 256),
        CloseCh: make(chan struct{}),
    }
    
    // 加入连接池
    h.manager.AddConn(userId, conn)
    
    // 启动读写循环
    go conn.readLoop()
    go conn.writeLoop()
}

// 2. 推送消息
func (m *Manager) PushToUser(userId int64, message []byte) error {
    val, ok := m.Connections.Load(userId)
    if !ok {
        return errors.New("用户不在线")
    }
    conn := val.(*Conn)
    
    // 非阻塞发送，超时 5 秒
    select {
    case conn.SendCh <- message:
        return nil
    case <-time.After(5 * time.Second):
        return errors.New("发送超时")
    }
}

// 3. 关闭连接
func (c *Conn) Close() {
    if c.IsClosed.Swap(true) {
        return  // 已经关闭过了
    }
    close(c.CloseCh)  // 通知 writeLoop 退出
    c.WsConn.Close()
    // Manager 会自动移除连接
}
```

### 1.4 JWT 双 Token 认证机制

#### 为什么需要双 Token？

| Token 类型 | 有效期 | 存储位置 | 用途 |
|-----------|-------|---------|------|
| **AccessToken** | 15 分钟 | 内存/LocalStorage | 每次 API 调用携带，验证身份 |
| **RefreshToken** | 7 天 | HttpOnly Cookie | 用于刷新 AccessToken |

**核心优势：**
1. **安全性**：AccessToken 短期，即使泄漏影响也有限
2. **用户体验**：RefreshToken 长期，用户不需要频繁登录
3. **可撤销性**：服务端可以在 Redis 中删除 RefreshToken，强制下线

#### Token 生成逻辑

```go
// auth-service/internal/logic/generate_token_logic.go
func (l *GenerateTokenLogic) GenerateToken(userId int64, deviceId string) (string, string, error) {
    // 1. 生成 Access Token
    accessClaims := jwt.MapClaims{
        "user_id":   userId,
        "device_id": deviceId,
        "exp":       time.Now().Add(15 * time.Minute).Unix(),
        "iat":       time.Now().Unix(),
    }
    accessToken := jwt.NewWithClaims(jwt.SigningMethodHS256, accessClaims)
    accessTokenString, _ := accessToken.SignedString([]byte(l.svcCtx.Config.JwtSecret))
    
    // 2. 生成 Refresh Token
    refreshClaims := jwt.MapClaims{
        "user_id": userId,
        "exp":     time.Now().Add(7 * 24 * time.Hour).Unix(),
    }
    refreshToken := jwt.NewWithClaims(jwt.SigningMethodHS256, refreshClaims)
    refreshTokenString, _ := refreshToken.SignedString([]byte(l.svcCtx.Config.JwtSecret))
    
    // 3. 存储 Session 到 Redis
    session := &model.Session{
        Uid:          userId,
        DeviceId:     deviceId,
        AccessToken:  accessTokenString,
        RefreshToken: refreshTokenString,
        ExpiresAt:    time.Now().Add(7 * 24 * time.Hour),
    }
    l.svcCtx.SessionModel.Save(session)
    
    return accessTokenString, refreshTokenString, nil
}
```

#### Token 刷新流程

```go
// auth-service/internal/logic/refresh_token_logic.go
func (l *RefreshTokenLogic) RefreshToken(req *auth.RefreshTokenRequest) (*auth.RefreshTokenResponse, error) {
    // 1. 验证 Refresh Token
    claims, err := jwt.Parse(req.RefreshToken)
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "Token 无效")
    }
    
    userId := claims["user_id"].(int64)
    
    // 2. 检查 Redis 中是否存在该 Session
    session, err := l.svcCtx.SessionModel.FindByRefreshToken(req.RefreshToken)
    if err != nil {
        return nil, status.Error(codes.NotFound, "会话已过期")
    }
    
    // 3. 生成新的 Access Token（Refresh Token 不变）
    newAccessToken, _ := l.GenerateAccessToken(userId, session.DeviceId)
    
    // 4. 更新 Session
    session.AccessToken = newAccessToken
    l.svcCtx.SessionModel.Update(session)
    
    return &auth.RefreshTokenResponse{
        AccessToken: newAccessToken,
        ExpiresIn:   900, // 15 分钟
    }, nil
}
```

#### 多设备登录管理

**核心思想**：同一账号在不同设备登录，生成不同的 Session

```go
// sessions 表结构
CREATE TABLE sessions (
    uid BIGINT,
    device_id VARCHAR(100),
    access_token VARCHAR(500),
    refresh_token VARCHAR(500),
    UNIQUE KEY uk_uid_device (uid, device_id)  -- 同账号同设备只能有一个会话
);
```

**场景演示：**
1. 用户在手机登录：`device_id = "iPhone-12345"`
2. 用户在电脑登录：`device_id = "Chrome-67890"`
3. 两个 Session 并存，互不影响
4. 如果手机重新登录，旧的手机 Session 会被覆盖（踢出旧设备）

### 1.5 防骚扰机制（黑名单系统）

#### 业务场景

用户 A 拉黑用户 B 后：
- B 无法给 A 发消息
- B 无法看到 A 的在线状态
- B 无法再次添加 A 为好友

#### 实现逻辑

```go
// chat-service/internal/logic/send_message_logic.go
func (l *SendMessageLogic) SendMessage(req *chat.SendMessageRequest) (*chat.SendMessageResponse, error) {
    // 1. 检查是否被拉黑（调用 friend-service）
    blocked, err := l.svcCtx.FriendRpc.IsBlocked(context.Background(), &friend.IsBlockedRequest{
        UserId:  req.ToUserId,   // 接收者
        CheckId: req.FromUserId, // 发送者
    })
    
    if blocked.IsBlocked {
        return nil, status.Error(codes.PermissionDenied, "对方已将您拉黑")
    }
    
    // 2. 正常发送消息
    // ...
}
```

**friend-service 实现：**
```go
// friend-service/internal/logic/is_blocked_logic.go
func (l *IsBlockedLogic) IsBlocked(req *friend.IsBlockedRequest) (*friend.IsBlockedResponse, error) {
    // 查询 blocked_users 表
    var count int64
    l.svcCtx.DB.Model(&model.BlockedUser{}).
        Where("user_id = ? AND blocked_id = ?", req.UserId, req.CheckId).
        Count(&count)
    
    return &friend.IsBlockedResponse{
        IsBlocked: count > 0,
    }, nil
}
```

#### 性能优化

**问题**：每次发消息都要查数据库，高并发下会成为瓶颈

**解决方案**：Redis 缓存黑名单关系
```go
// 缓存 Key: block:{userId}:{blockedId}
key := fmt.Sprintf("block:%d:%d", userId, blockedId)
exists, err := l.svcCtx.RedisClient.Exists(context.Background(), key).Result()
if err == nil && exists == 1 {
    return &friend.IsBlockedResponse{IsBlocked: true}, nil
}

// 未命中缓存，查数据库
// ...

// 写入缓存（TTL = 10 分钟）
if count > 0 {
    l.svcCtx.RedisClient.Set(context.Background(), key, "1", 10*time.Minute)
}
```

### 1.6 消息幂等性保证

#### 为什么需要幂等性？

**场景**：客户端网络抖动，重复发送同一条消息
```
客户端发送：你好
↓
网络超时，未收到响应
↓
客户端重试：你好（重复）
```

**问题**：接收者会收到两条相同的消息

#### 解决方案：UUID 作为消息 ID

```go
// chat-service/internal/logic/send_message_logic.go
msg := &Message{
    Id:         uuid.New().String(),  // 客户端生成或服务端生成
    FromUserId: req.FromUserId,
    ToUserId:   req.ToUserId,
    Content:    req.Content,
    CreatedAt:  time.Now(),
}

// 存储到 MySQL 时，主键为 Id（UUID）
// 如果重复插入，会触发主键冲突，直接忽略
l.svcCtx.DB.Create(msg)
```

**进阶方案**：客户端生成 UUID
```javascript
// 客户端代码
const messageId = uuidv4();
ws.send(JSON.stringify({
    id: messageId,
    content: "你好"
}));

// 重试时，使用相同的 messageId
```

**服务端去重逻辑**：
```go
// 检查消息是否已存在
var count int64
l.svcCtx.DB.Model(&Message{}).Where("id = ?", req.MessageId).Count(&count)
if count > 0 {
    return &chat.SendMessageResponse{
        MessageId: req.MessageId,
        Success:   true,  // 直接返回成功，避免重复处理
    }, nil
}
```

### 2.1 数据库查询优化

#### 索引设计

**聊天记录查询（最频繁操作）**
```sql
-- 查询 A 和 B 之间的聊天记录
SELECT * FROM messages 
WHERE (from_user_id = A AND to_user_id = B)
   OR (from_user_id = B AND to_user_id = A)
ORDER BY created_at DESC
LIMIT 20;

-- 索引设计
CREATE INDEX idx_conversation ON messages(from_user_id, to_user_id, created_at);
```

**查询计划分析**：
```sql
EXPLAIN SELECT * FROM messages 
WHERE from_user_id = 123 AND to_user_id = 456 
ORDER BY created_at DESC LIMIT 20;

-- 结果
type: ref
key: idx_conversation
rows: 150  -- 只扫描 150 行，而不是全表
Extra: Using index condition
```

#### 分页优化

**传统分页问题**：
```sql
-- 第 100 页（OFFSET 2000）
SELECT * FROM messages WHERE to_user_id = 123 
ORDER BY created_at DESC LIMIT 20 OFFSET 2000;

-- 问题：需要扫描 2020 行，然后丢弃前 2000 行
```

**游标分页（更高效）**：
```sql
-- 第一页
SELECT * FROM messages WHERE to_user_id = 123 
ORDER BY created_at DESC LIMIT 20;

-- 记录最后一条消息的 created_at = '2026-02-03 10:00:00'

-- 第二页
SELECT * FROM messages WHERE to_user_id = 123 
AND created_at < '2026-02-03 10:00:00'
ORDER BY created_at DESC LIMIT 20;

-- 性能提升：始终只扫描 20 行
```

### 2.2 Redis 缓存优化

#### 缓存穿透防护

**场景**：查询一个不存在的用户 ID，绕过缓存直接打到数据库

**解决方案**：缓存空值
```go
func (l *GetUserLogic) GetUser(userId int64) (*User, error) {
    // 1. 查 Redis
    key := fmt.Sprintf("user:%d", userId)
    val, err := l.svcCtx.RedisClient.Get(context.Background(), key).Result()
    
    if err == redis.Nil {
        // 2. 查数据库
        user, err := l.svcCtx.UserModel.FindById(userId)
        if err != nil {
            // 3. 缓存空值（TTL = 5 分钟）
            l.svcCtx.RedisClient.Set(context.Background(), key, "null", 5*time.Minute)
            return nil, err
        }
        
        // 4. 缓存正常值
        l.svcCtx.RedisClient.Set(context.Background(), key, toJSON(user), 10*time.Minute)
        return user, nil
    }
    
    if val == "null" {
        return nil, errors.New("用户不存在")
    }
    
    return fromJSON(val), nil
}
```

#### 缓存雪崩防护

**场景**：大量缓存同时过期，流量瞬间打到数据库

**解决方案**：过期时间加随机值
```go
// 不要所有缓存都是 10 分钟
ttl := 10*time.Minute + time.Duration(rand.Intn(60))*time.Second
l.svcCtx.RedisClient.Set(context.Background(), key, value, ttl)
```

### 2.3 Kafka 性能调优

#### 批量发送

```go
// 不好的做法：每条消息单独发送
for _, msg := range messages {
    producer.WriteMessages(context.Background(), kafka.Message{...})
}

// 好的做法：批量发送
var kafkaMessages []kafka.Message
for _, msg := range messages {

    })
}
producer.WriteMessages(context.Background(), kafkaMessages...)
```

**性能提升**：吞吐量提高 10 倍以上

#### 消费者并发优化

```go
// ws-gateway/internal/kafka/consumer.go
func StartChatConsumer(manager *ws.Manager) {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers:         []string{"localhost:9092"},
        Topic:           "im-chat-topic",
        GroupID:         "ws-gateway-group",
        MinBytes:        10e3,  // 10KB，攒够才拉取
        MaxBytes:        10e6,  // 10MB
        CommitInterval:  time.Second,  // 每秒提交一次 offset
    })
    
    // 启动多个 goroutine 消费
    for i := 0; i < 10; i++ {
        go func() {
            for {
                msg, _ := reader.FetchMessage(context.Background())
                // 处理消息
                manager.PushToUser(userId, msg.Value)
                // 提交 offset
                reader.CommitMessages(context.Background(), msg)
            }
        }()
    }
}
```

## 三、难点突破

### 3.1 消息顺序性保证

**挑战**：多个服务实例并发处理，如何保证消息顺序？

**解决方案**：
1. **Kafka 分区键**：同一用户消息路由到同一分区
2. **单分区单消费者**：一个分区只被一个消费者消费
3. **串行处理**：消费者内部不使用并发处理同一用户的消息

```go
// Kafka 配置
Topic: im-chat-topic
Partitions: 16  // 16 个分区
Key: ToUserId   // 用户 ID 作为分区键

// 哈希算法
partition := hash(ToUserId) % 16

// 结果：同一用户的所有消息都在同一分区
// 分区内消息严格有序
```

### 3.2 在线状态管理

**挑战**：如何判断用户是否在线？

**方案一：WebSocket 连接 = 在线（不准确）**
- 问题：连接断开到检测到有延迟（心跳间隔 30 秒）

**方案二：Redis + TTL（本项目采用）**
```go
// 用户建立 WebSocket 连接时
l.svcCtx.RedisClient.Set(context.Background(), 
    fmt.Sprintf("user:online:%d", userId), 
    "1", 
    300*time.Second,  // 5 分钟
)

// 心跳刷新 TTL
ticker := time.NewTicker(30 * time.Second)
for range ticker.C {
    l.svcCtx.RedisClient.Expire(context.Background(), key, 300*time.Second)
}

// 查询在线状态
func IsOnline(userId int64) bool {
    val, _ := redis.Get(fmt.Sprintf("user:online:%d", userId)).Result()
    return val == "1"
}
```

**方案三：多端在线状态**
```go
// Redis Hash 结构
HSET user:online:123 "device:phone" "1"
HSET user:online:123 "device:pc" "1"

// 查询：只要有一个设备在线就是在线
HLEN user:online:123 > 0
```

```go
func OnUserConnect(userId int64) {
```

## 四、面试高频问题解答

### Q1: 如何保证消息不丢失？

**三层保障**：
1. **Kafka 持久化**：副本因子 = 3，消息写入多个 Broker
2. **Producer ACK**：`acks=all`，所有副本确认后才返回
3. **Consumer 手动提交**：处理成功后再提交 offset

### Q2: 如何处理消息积压？

**监控指标**：Kafka Lag（消费延迟）

**解决方案**：
1. **增加分区数**：提高并行度
2. **增加消费者实例**：水平扩展
3. **批量消费**：每次拉取更多消息

### Q3: WebSocket 如何扩展到多台机器？

**方案**：Redis Pub/Sub
```go
// 机器 A 收到消息，发布到 Redis
redis.Publish("ws:push", JSON.stringify({
    userId: 123,
    message: "你好"
}))

// 机器 B 订阅频道
redis.Subscribe("ws:push", func(msg) {
    // 如果用户连接在本机，推送
    if manager.HasConnection(msg.userId) {
        manager.PushToUser(msg.userId, msg.message)
    }
})
```

### Q4: 如何防止重复消费？

**幂等性设计**：
1. 消息 ID 使用 UUID
2. 数据库主键去重
3. Redis 去重（`SETNX`）
