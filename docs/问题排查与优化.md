## 二、常见性能瓶颈与优化

### 2.1 WebSocket 连接性能问题

#### 问题 1：连接数达到上限后无法建立新连接

**现象：**
```bash
# 客户端报错
WebSocket connection failed: Too many open files

# 服务端日志
accept tcp [::]:8080: accept4: too many open files in system
```

**根因分析：**
- Linux 系统默认文件描述符限制为 1024
- 每个 WebSocket 连接占用 1 个文件描述符
- 超过限制后无法创建新连接

**解决方案：**
```bash
# 1. 临时修改（重启失效）
ulimit -n 1000000

# 2. 永久修改 /etc/security/limits.conf
* soft nofile 1000000
* hard nofile 1000000

# 3. 修改系统参数 /etc/sysctl.conf
fs.file-max = 1000000
fs.nr_open = 1000000

# 4. 重启生效
sudo sysctl -p
sudo reboot

# 5. 验证
ulimit -n
# 输出：1000000
```

**代码层面优化：**
```go
// 设置 WebSocket 连接池大小限制
type Manager struct {
    MaxConnections int
    CurrentCount   atomic.Int64
}

func (m *Manager) AddConn(userId int64, conn *Conn) error {
    if m.CurrentCount.Load() >= int64(m.MaxConnections) {
        return errors.New("连接数已达上限")
    }
    
    m.Connections.Store(userId, conn)
    m.CurrentCount.Add(1)
    
    go conn.readLoop()
    go conn.writeLoop()
    return nil
}

func (m *Manager) RemoveConn(userId int64) {
    m.Connections.Delete(userId)
    m.CurrentCount.Add(-1)
}
```

#### 问题 2：内存泄漏导致 OOM

**现象：**
```bash
# 服务器内存使用持续上升
free -h
              total        used        free
Mem:           8.0G        7.8G        200M

# 进程被 OOM Killer 杀死
dmesg | grep -i "killed process"
Out of memory: Killed process 12345 (ws-gateway)
```

**排查步骤：**
```bash
# 1. 使用 pprof 分析内存
curl http://localhost:6060/debug/pprof/heap > heap.prof
go tool pprof heap.prof

# 2. 查看内存分配 top 10
(pprof) top10
Showing nodes accounting for 2GB, 95% of 2.1GB total
      flat  flat%   sum%        cum   cum%
    1.2GB 57.14% 57.14%     1.2GB 57.14%  ws.(*Conn).readLoop
    500MB 23.81% 80.95%     500MB 23.81%  ws.(*Manager).AddConn
    300MB 14.29% 95.24%     300MB 14.29%  encoding/json.Unmarshal

# 3. 查看 goroutine 数量
curl http://localhost:6060/debug/pprof/goroutine?debug=1
goroutine profile: total 100000  ← 异常，正常应该 < 10000
```

**根因分析：**
- WebSocket 连接关闭后，goroutine 没有退出
- SendCh 缓冲区未清空，导致内存无法释放
- 闭包引用导致对象无法被 GC

**解决方案：**
```go
// 优化前（有问题）
func (c *Conn) Close() {
    c.WsConn.Close()
    // 问题：goroutine 可能还在运行，无法回收
}

// 优化后（正确）
func (c *Conn) Close() {
    if c.IsClosed.Swap(true) {
        return  // 避免重复关闭
    }
    
    close(c.CloseCh)     // 通知 goroutine 退出
    c.WsConn.Close()
    
    // 清空缓冲区
    for len(c.SendCh) > 0 {
        <-c.SendCh
    }
    close(c.SendCh)
}

func (c *Conn) writeLoop() {
    defer func() {
        // 确保 panic 不会导致 goroutine 泄漏
        if r := recover(); r != nil {
            log.Printf("writeLoop panic: %v", r)
        }
    }()
    
    for {
        select {
        case msg := <-c.SendCh:
            c.WsConn.WriteMessage(websocket.TextMessage, msg)
        case <-c.CloseCh:
            return  // 收到关闭信号，退出 goroutine
        }
    }
}
```

### 2.2 Kafka 消费延迟问题

#### 问题 3：消息积压导致延迟增加

**现象：**
```bash
# 查看 Consumer Lag
kafka-consumer-groups.sh --describe --group ws-gateway-group --bootstrap-server localhost:9092

GROUP           TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
ws-gateway      im-chat-topic   0          50000           100000          50000  ← 积压 5 万条
ws-gateway      im-chat-topic   1          48000           98000           50000
```

**根因分析：**
1. **消费速度 < 生产速度**
   - 生产速度：1000 msg/s
   - 消费速度：200 msg/s（单实例）
   - 差值：800 msg/s 持续积压

2. **消费逻辑耗时长**
   ```go
   for {
       msg := reader.ReadMessage()      // 5ms
       pushToWebSocket(msg)              // 10ms
       db.Create(&Message{})             // 30ms ← 瓶颈
       reader.CommitMessages(msg)        // 5ms
   }
   // 单条消息耗时：50ms
   // 单线程 QPS：1000 / 50 = 20 msg/s
   ```

**解决方案 1：水平扩展消费者**
```bash
# Docker Compose 扩容
docker-compose scale ws-gateway=3

# 效果：
# 1 个实例：200 msg/s
# 3 个实例：600 msg/s
# Lag 逐渐降低
```

**解决方案 2：异步写数据库**
```go
// 优化前：同步写（阻塞）
func processMessage(msg kafka.Message) {
    pushToWebSocket(msg)  // 10ms
    db.Create(&Message{}) // 30ms ← 阻塞
}

// 优化后：异步写（不阻塞）
var dbWriteQueue = make(chan *Message, 10000)

// 消费者协程
func processMessage(msg kafka.Message) {
    pushToWebSocket(msg)  // 10ms
    
    // 投递到队列，不等待
    select {
    case dbWriteQueue <- &Message{}:
    default:
        log.Warn("写队列已满，消息可能丢失")
    }
}

// 独立的数据库写入协程
func dbWriter() {
    batch := make([]*Message, 0, 100)
    ticker := time.NewTicker(time.Second)
    
    for {
        select {
        case msg := <-dbWriteQueue:
            batch = append(batch, msg)
            
            if len(batch) >= 100 {
                db.CreateInBatches(batch, 100)  // 批量写入
                batch = batch[:0]
            }
            
        case <-ticker.C:
            if len(batch) > 0 {
                db.CreateInBatches(batch, len(batch))
                batch = batch[:0]
            }
        }
    }
}

// 启动多个写入协程
for i := 0; i < 5; i++ {
    go dbWriter()
}
```

**效果对比：**
| 方案 | 消息处理耗时 | 单实例 QPS | 3 实例 QPS |
|------|------------|----------|-----------|
| 优化前 | 50ms | 20 msg/s | 60 msg/s |
| 优化后 | 10ms | 100 msg/s | 300 msg/s |

**解决方案 3：增加 Kafka 分区**
```bash
# 查看当前分区数
kafka-topics.sh --describe --topic im-chat-topic --bootstrap-server localhost:9092
Topic: im-chat-topic    PartitionCount: 4

# 增加到 16 个分区（不可逆）
kafka-topics.sh --alter --topic im-chat-topic --partitions 16 --bootstrap-server localhost:9092

# 效果：
# 4 分区 + 3 消费者：每个消费者负责 1-2 个分区
# 16 分区 + 3 消费者：分区利用不充分
# 建议：分区数 = 消费者数的 2-4 倍
```

### 2.3 数据库查询性能问题

#### 问题 4：聊天记录查询慢

**现象：**
```sql
-- 查询 A 和 B 的聊天记录（耗时 5 秒）
SELECT * FROM messages 
WHERE (from_user_id = 123 AND to_user_id = 456)
   OR (from_user_id = 456 AND to_user_id = 123)
ORDER BY created_at DESC 
LIMIT 20;

-- EXPLAIN 分析
type: ALL  ← 全表扫描
rows: 5000000  ← 扫描 500 万行
```

**根因分析：**
- 没有合适的索引
- OR 条件导致索引失效
- 数据量大（messages 表 500 万条记录）

**解决方案 1：添加组合索引**
```sql
-- 创建索引
CREATE INDEX idx_conversation ON messages(from_user_id, to_user_id, created_at);

-- 再次 EXPLAIN
type: range
key: idx_conversation
rows: 150  ← 只扫描 150 行
Extra: Using index condition

-- 查询时间从 5 秒降到 50ms
```

**解决方案 2：改写查询（避免 OR）**
```sql
-- 优化前
WHERE (from_user_id = 123 AND to_user_id = 456)
   OR (from_user_id = 456 AND to_user_id = 123)

-- 优化后：使用 UNION
(SELECT * FROM messages WHERE from_user_id = 123 AND to_user_id = 456)
UNION ALL
(SELECT * FROM messages WHERE from_user_id = 456 AND to_user_id = 123)
ORDER BY created_at DESC LIMIT 20;

-- 效果：两个查询都能用上索引
```

**解决方案 3：分表**
```go
// 按用户 ID 哈希分表（16 张表）
func getTableName(userId int64) string {
    tableIndex := userId % 16
    return fmt.Sprintf("messages_%02d", tableIndex)
}

// 查询时路由到对应分表
func GetChatHistory(userA, userB int64) ([]*Message, error) {
    // 确定在哪张表
    table := getTableName(userA)
    
    var messages []*Message
    db.Table(table).
        Where("(from_user_id = ? AND to_user_id = ?) OR (from_user_id = ? AND to_user_id = ?)",
            userA, userB, userB, userA).
        Order("created_at DESC").
        Limit(20).
        Find(&messages)
    
    return messages, nil
}

// 效果：单表数据量从 500 万降到 30 万，查询速度提升 10 倍
```

#### 问题 5：未读消息统计慢

**现象：**
```sql
-- 统计用户 123 的未读消息数（耗时 2 秒）
SELECT COUNT(*) FROM messages 
WHERE to_user_id = 123 AND status = 0;

-- EXPLAIN 分析
type: ref
key: idx_to_user
rows: 100000  ← 扫描 10 万行
```

**根因分析：**
- COUNT(*) 需要扫描所有未读消息
- 用户未读消息可能很多（10 万条）

**解决方案 1：Redis 计数器**
```go
// 发送消息时增加计数
func SendMessage(toUserId int64, msg *Message) error {
    // 1. 发送到 Kafka
    kafka.Produce(msg)
    
    // 2. Redis 计数器 +1
    redis.Incr(fmt.Sprintf("unread:%d", toUserId))
    
    return nil
}

// 用户读取消息时减少计数
func AckMessage(userId int64, messageId string) error {
    // 1. 更新数据库
    db.Model(&Message{}).Where("id = ?", messageId).Update("status", 1)
    
    // 2. Redis 计数器 -1
    redis.Decr(fmt.Sprintf("unread:%d", userId))
    
    return nil
}

// 查询未读数（从 Redis）
func GetUnreadCount(userId int64) (int64, error) {
    count, _ := redis.Get(fmt.Sprintf("unread:%d", userId)).Int64()
    return count, nil
}

// 效果：查询时间从 2 秒降到 1ms
```

**解决方案 2：定期全量同步**
```go
// 每小时全量同步一次（修正 Redis 计数偏差）
func syncUnreadCount() {
    for {
        time.Sleep(time.Hour)
        
        // 从数据库查询所有用户的未读数
        var results []struct {
            ToUserId int64
            Count    int64
        }
        db.Model(&Message{}).
            Select("to_user_id, COUNT(*) as count").
            Where("status = 0").
            Group("to_user_id").
            Find(&results)
        
        // 写入 Redis
        for _, r := range results {
            redis.Set(fmt.Sprintf("unread:%d", r.ToUserId), r.Count, 0)
        }
    }
}
```

### 2.4 Redis 缓存问题

#### 问题 6：缓存穿透

**现象：**
```bash
# 大量查询不存在的用户
GET /api/users/999999999  # 用户不存在
GET /api/users/999999998  # 用户不存在
# ...（每秒 1000 次）

# MySQL 慢查询日志暴增
slow_query.log:
SELECT * FROM users WHERE id = 999999999;  (0.5s)
SELECT * FROM users WHERE id = 999999998;  (0.5s)
```

**根因分析：**
- 攻击者故意查询不存在的数据
- Redis 缓存 miss，请求直接打到数据库
- 数据库压力剧增

**解决方案：缓存空值**
```go
func GetUser(userId int64) (*User, error) {
    // 1. 查 Redis
    key := fmt.Sprintf("user:%d", userId)
    val, err := redis.Get(key).Result()
    
    if err == redis.Nil {
        // 2. Redis 未命中，查数据库
        user, err := db.FindById(userId)
        if err != nil {
            // 3. 数据库也没有，缓存空值（TTL = 5 分钟）
            redis.Set(key, "null", 5*time.Minute)
            return nil, err
        }
        
        // 4. 缓存正常值（TTL = 30 分钟）
        redis.Set(key, toJSON(user), 30*time.Minute)
        return user, nil
    }
    
    // 5. 判断是否为空值
    if val == "null" {
        return nil, errors.New("用户不存在")
    }
    
    return fromJSON(val), nil
}

// 效果：攻击请求被 Redis 挡住，不会打到数据库
```

#### 问题 7：缓存雪崩

**现象：**
```bash
# 凌晨 3 点，缓存集体过期
# Redis 监控
15:00:00  200 queries/s
15:00:01  500 queries/s
15:00:02  10000 queries/s  ← 突然暴增

# MySQL 连接数暴增
SHOW PROCESSLIST;
1200 rows (max_connections = 1000)  ← 连接池耗尽
```

**根因分析：**
- 所有用户信息缓存设置相同 TTL（30 分钟）
- 同一时刻过期，请求同时打到数据库
- 数据库无法承受瞬时高并发

**解决方案：随机 TTL**
```go
// 优化前：固定 30 分钟
redis.Set(key, value, 30*time.Minute)

// 优化后：30 分钟 + 随机 0-5 分钟
randomOffset := time.Duration(rand.Intn(300)) * time.Second
ttl := 30*time.Minute + randomOffset
redis.Set(key, value, ttl)

// 效果：缓存过期时间分散，不会集中失效
```

### 3.3 故障案例与教训

#### 案例 1：Redis 主节点宕机

**事故经过：**
- 22:30 Redis 主节点因磁盘故障宕机
- 22:31 所有服务无法连接 Redis，报错 `connection refused`
- 22:32 用户无法登录、无法查看好友列表
- 22:35 运维切换到 Redis 从节点
- 22:40 服务恢复

**影响：**
- 故障时长：10 分钟
- 影响用户：1000+ 在线用户无法使用
- 数据丢失：无（Redis 持久化生效）

**教训：**
- 应该使用 Redis 哨兵模式，自动故障转移
- 客户端应该配置重试机制
- 应该有降级方案（Redis 不可用时查数据库）

**改进方案：**
```go
// 使用 Redis 哨兵
rdb := redis.NewFailoverClient(&redis.FailoverOptions{
    MasterName:    "mymaster",
    SentinelAddrs: []string{"sentinel1:26379", "sentinel2:26379"},
})

// 增加降级逻辑
func GetUser(userId int64) (*User, error) {
    // 尝试从 Redis 获取
    user, err := getUserFromRedis(userId)
    if err == nil {
        return user, nil
    }
    
    // Redis 失败，降级到数据库
    log.Warn("Redis 不可用，降级到数据库")
    return getUserFromDB(userId)
}
```

#### 案例 2：Kafka 消费积压

**事故经过：**
- 16:00 运营活动开始，消息量暴增 10 倍
- 16:05 用户反馈消息延迟 10 秒以上
- 16:10 检查 Kafka Lag，发现积压 50 万条
- 16:15 紧急扩容 WS Gateway 从 1 个实例到 5 个
- 16:30 Lag 降为 0，服务恢复

**影响：**
- 故障时长：30 分钟
- 消息延迟：最高 30 秒
- 用户投诉：100+ 条

**教训：**
- 应该提前压测，评估系统容量
- 应该配置自动扩容（K8s HPA）
- 应该监控 Kafka Lag，达到阈值自动告警

**改进方案：**
```yaml
# Kubernetes HPA（自动扩容）
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ws-gateway-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ws-gateway
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: External
    external:
      metric:
```

## 四、面试中如何回答优化问题

### 回答模板

**面试官**：你是如何优化系统性能的？

**回答结构：**
1. **发现问题**：通过什么方式发现的（监控、日志、用户反馈）
2. **定位原因**：如何排查的（pprof、慢查询日志、EXPLAIN）
3. **解决方案**：具体做了什么优化
4. **优化效果**：用数据说话（响应时间从 X 降到 Y）
5. **经验总结**：学到了什么

**示例回答：**

"在项目运行过程中，我发现 WebSocket 服务的内存使用持续增长，最终导致 OOM。

**排查过程**：我使用 Go 的 pprof 工具分析内存，发现大量 goroutine 泄漏。通过 `debug/pprof/goroutine` 发现数量异常（10 万个），正常应该只有几千个。

**定位原因**：通过查看代码和 goroutine 堆栈，发现是 WebSocket 连接关闭后，readLoop 和 writeLoop 两个 goroutine 没有正常退出，导致相关对象无法被 GC 回收。

**解决方案**：
1. 使用 `CloseCh` 通道通知 goroutine 退出
2. 使用 `atomic.Bool` 避免重复关闭连接
3. 在 goroutine 中添加 `defer recover()` 防止 panic 导致泄漏

**优化效果**：
- goroutine 数量从 10 万降到 5000
- 内存使用从 2GB 降到 200MB
- 服务稳定运行 7 天无重启

**经验总结**：
- 长期运行的 goroutine 一定要有退出机制
- 使用 pprof 定期检查内存和 goroutine
- 生产环境要开启 pprof 接口，方便排查问题"
