# IM即时通讯系统 - 面试准备速查表

## 一、30秒项目介绍（电梯演讲）

"这是我开发的一个**基于 Go 语言的微服务架构 IM 即时通讯系统**。

**技术栈**：采用 go-zero 框架，通过 gRPC 实现服务间通信，使用 Kafka 作为消息队列实现异步分发，WebSocket 提供实时推送，Redis 管理会话和在线状态，MySQL 做数据持久化。

**核心功能**：用户注册登录、JWT 双 Token 认证、好友管理（添加/删除/拉黑）、实时单聊、消息已读回执。

**技术亮点**：
1. Kafka 分区键设计保证消息顺序性
2. WebSocket 连接池 + 读写分离，单机支持 10 万并发
3. 多级缓存架构，API 响应时间 P99 < 200ms

整个系统支持 Docker Compose 一键部署，具备生产级别的高可用和监控能力。"

## 二、核心技术点速查

### 微服务架构

**拆分原则**：
- Auth Service → 认证授权
- User Service → 用户管理
- Chat Service → 消息处理
- Friend Service → 好友关系
- WS Gateway → 长连接管理

**优势**：独立部署、故障隔离、水平扩展

---

### Kafka 消息队列

**三大作用**：
1. **削峰填谷** - 缓冲瞬时高并发
2. **异步解耦** - 发送者和接收者分离
3. **持久化** - 副本因子=3，消息不丢失

**关键配置**：
```yaml
Topic: im-chat-topic
Partitions: 16
Key: recipient_id  # 保证消息顺序
```

---

### WebSocket 长连接

**连接管理**：
- `sync.Map` 存储连接池（并发安全）
- 读写分离（独立 goroutine）
- 心跳保活（30 秒 PING）
- 缓冲队列（256 容量防阻塞）

**性能数据**：
- 单机并发：10 万连接
- 内存占用：200MB
- 延迟：10-50ms

---

### JWT 双 Token

| Token | 有效期 | 用途 | 可撤销 |
|-------|-------|------|-------|
| Access | 15分钟 | API调用 | 否 |
| Refresh | 7天 | 刷新Access | 是 |

**安全性**：
- bcrypt 密码加密（成本因子12）
- 设备指纹识别（device_id）
- Redis 存储 Session，支持强制下线

## 三、高频面试问题快速应答

### Q: 如何保证消息不丢失？

**三层保障**：
1. Kafka 持久化（副本因子=3）
2. Producer ACK=all（所有副本确认）
3. Consumer 手动提交（处理成功再提交 offset）

---

### Q: 如何保证消息顺序？

**Kafka 分区键**：
- Key = recipient_id
- 同一用户消息路由到同一分区
- 分区内严格有序

**注意**：只保证单用户有序，不保证全局有序（不需要）

---

### Q: WebSocket 如何水平扩展？

**Redis Pub/Sub 方案**：
```
1. 机器 A 收到 Kafka 消息
2. 用户不在机器 A，发布到 Redis
3. 机器 B 订阅 Redis，收到消息
4. 推送给连接在机器 B 的用户
```

**优化**：一致性哈希让大部分消息本地推送

---

### Q: 如何处理高并发？

**架构层面**：
- 微服务水平扩展（3-5 个实例）
- MySQL 主从读写分离
- Redis 哨兵模式
- Kafka 16 分区

**代码层面**：
- 连接池（MySQL、Redis）
- 异步写数据库（不阻塞消息推送）
- 批量查询（避免 N+1）

---

### Q: 数据库如何优化？

**索引设计**：
```sql
-- 聊天记录查询
INDEX idx_conversation (from_user_id, to_user_id, created_at)

-- 未读消息统计
INDEX idx_to_user (to_user_id, status, created_at)
```

**分表策略**：
- 按用户 ID 哈希分 16 张表
- 单表 500万 → 每表 30万
- 查询速度提升 10 倍

**缓存策略**：
- Redis 缓存用户信息（TTL=30min）
- Redis 计数器统计未读数

## 四、项目难点与解决方案

### 难点 1：WebSocket goroutine 泄漏

**问题**：连接关闭后 goroutine 不退出，内存泄漏

**解决**：
- 使用 `CloseCh` 通知 goroutine 退出
- `atomic.Bool` 避免重复关闭
- pprof 监控 goroutine 数量

**效果**：goroutine 从 10 万降到 5000

---

### 难点 2：Kafka 消费延迟

**问题**：高峰期消息积压 50 万条，延迟 10 秒

**解决**：
1. 水平扩容（1→5 个消费者）
2. 异步写数据库（不阻塞推送）
3. 批量消费（一次拉 100 条）

**效果**：延迟从 10 秒降到 50ms

---

### 难点 3：数据库查询慢

**问题**：聊天记录查询 5 秒

**解决**：
1. 添加组合索引
2. 避免 OR 查询（改用 UNION）
3. 按用户分表

**效果**：查询从 5 秒降到 50ms

### 优化案例

**案例 1：缓存穿透**
- 问题：大量查询不存在的用户
- 解决：缓存空值（TTL=5min）
- 效果：数据库压力降低 90%

**案例 2：缓存雪崩**
- 问题：缓存同时过期，数据库被打垮
- 解决：TTL 加随机值（避免集中失效）
- 效果：数据库连接数从 1200 降到 100
