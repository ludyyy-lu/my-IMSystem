# IM即时通讯系统 - 架构设计文档

## 一、项目概述

### 1.1 项目背景与目标

本项目是一个基于 **Go 语言** 和 **微服务架构** 构建的高性能即时通讯系统。项目旨在构建一个功能完善、架构清晰、具有企业级水平的 IM 后端系统,涵盖用户认证、好友管理、实时聊天、消息持久化等核心功能。

**核心目标：**
- ✅ 支持百万级在线用户并发
- ✅ 消息实时推送，延迟 < 100ms
- ✅ 微服务架构，服务独立部署、水平扩展
- ✅ 消息可靠性保证，支持离线消息推送
- ✅ 高可用架构设计，单点故障自动切换

### 1.2 技术选型理由

| 技术组件 | 选型 | 选择理由 |
|---------|------|---------|
| **编程语言** | Go 1.23+ | 高并发性能优异、协程模型适合 IO 密集型场景、编译型语言部署简单 |
| **微服务框架** | go-zero | 阿里云开源框架，内置服务发现、熔断限流、链路追踪，生态成熟 |
| **RPC 协议** | gRPC + Protobuf | 高性能二进制序列化、强类型定义、支持流式通信 |
| **实时通信** | WebSocket | 全双工通信、低延迟推送、浏览器原生支持 |
| **消息队列** | Kafka | 高吞吐量、持久化保证、分区机制支持水平扩展 |
| **缓存系统** | Redis | 高性能 KV 存储、支持多种数据结构、适合会话管理和在线状态 |
| **数据库** | MySQL 8.0 | 成熟稳定、支持事务、丰富的索引类型 |
| **ORM 框架** | GORM | Go 生态最流行 ORM、链式调用、自动迁移 |
| **服务发现** | Etcd | 强一致性、高可用、go-zero 原生支持 |
| **容器化** | Docker Compose | 快速部署、环境一致性、易于管理依赖服务 |

### 2.2 核心服务模块

#### 2.2.1 Auth Service (认证服务)

**职责：**
- 用户注册与登录验证
- JWT Token 生成与刷新
- 多设备会话管理
- Token 解析与权限校验

**核心技术点：**
- **JWT 双 Token 机制**：AccessToken (15分钟) + RefreshToken (7天)
- **设备指纹识别**：通过 device_id 区分不同终端
- **Session 存储**：Redis 存储会话信息，支持踢出设备
- **密码安全**：bcrypt 加密存储，防彩虹表攻击

**关键代码实现：**
```go
// auth-service/internal/logic/login_logic.go
func (l *LoginLogic) Login(req *auth.LoginRequest) (*auth.LoginResponse, error) {
    // 1. 查询用户并验证密码
    user, err := l.svcCtx.UserModel.FindByUsername(req.Username)
    if err := bcrypt.CompareHashAndPassword([]byte(user.Password), []byte(req.Password)); err != nil {
        return nil, status.Error(codes.Unauthenticated, "用户名或密码错误")
    }
    
    // 2. 生成双 Token
    accessToken := generateAccessToken(user.ID, req.DeviceId)
    refreshToken := generateRefreshToken(user.ID)
    
    // 3. 存储会话到 Redis
    session := &model.Session{
        Uid: user.ID,
        DeviceId: req.DeviceId,
        AccessToken: accessToken,
        RefreshToken: refreshToken,
        ExpiresAt: time.Now().Add(7 * 24 * time.Hour),
    }
    l.svcCtx.SessionModel.Save(session)
    
    return &auth.LoginResponse{
        AccessToken: accessToken,
        RefreshToken: refreshToken,
        ExpiresIn: 900, // 15分钟
    }, nil
}
```

#### 2.2.3 Chat Service (聊天服务)

**职责：**
- 单聊消息发送与接收
- 消息历史查询（分页）
- 消息状态管理（已读/未读）
- 消息回执确认（ACK）

**核心技术点：**
- **Kafka 解耦**：发送者和接收者通过 Kafka 异步通信
- **消息分区**：以 recipient_id 为 key，保证消息顺序
- **防骚扰机制**：发送前检查是否被拉黑
- **幂等性保证**：使用 UUID 作为消息 ID，防止重复消费

**消息流程：**
```go
// chat-service/internal/logic/send_message_logic.go
func (l *SendMessageLogic) SendMessage(req *chat.SendMessageRequest) (*chat.SendMessageResponse, error) {
    // 1. 检查是否被拉黑
    blocked, _ := l.svcCtx.FriendRpc.IsBlocked(context.Background(), &friend.IsBlockedRequest{
        UserId: req.ToUserId,
        CheckId: req.FromUserId,
    })
    if blocked.IsBlocked {
        return nil, status.Error(codes.PermissionDenied, "对方已将您拉黑")
    }
    
    // 2. 构造消息对象
    msg := &Message{
        Id: uuid.New().String(),
        FromUserId: req.FromUserId,
        ToUserId: req.ToUserId,
        Content: req.Content,
        MsgType: req.MsgType,
        Status: 0, // 未读
        CreatedAt: time.Now(),
    }
    
    // 3. 发送到 Kafka（异步）
    err := l.svcCtx.KafkaProducer.WriteMessages(context.Background(),
        kafka.Message{
            Topic: "im-chat-topic",
            Key: []byte(req.ToUserId), // 以接收者为分区键
            Value: toJSON(msg),
        },
    )
    
    // 4. 返回发送结果
    return &chat.SendMessageResponse{
        MessageId: msg.Id,
        Timestamp: msg.CreatedAt.Unix(),
        Success: true,
    }, nil
}
```

#### 2.2.4 Friend Service (好友服务)

**职责：**
- 好友添加请求与响应
- 好友列表查询
- 好友删除与拉黑
- 好友关系验证

**核心技术点：**
- **双向关系维护**：A添加B后，同时创建 A→B 和 B→A 两条记录
- **请求状态机**：待处理 → 已同意/已拒绝
- **黑名单优先级**：被拉黑后无法发送消息，即使是好友关系

**关键表设计：**
```sql
-- 好友关系表（双向）
CREATE TABLE friends (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    friend_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_friend (user_id, friend_id),
    INDEX idx_user (user_id)
);

-- 拉黑表
CREATE TABLE blocked_users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    blocked_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_blocked (user_id, blocked_id)

-- 好友请求表
CREATE TABLE friend_requests (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

```

#### 2.2.5 WS Gateway (WebSocket 网关)

**职责：**
- 管理 WebSocket 长连接
- 实时消息推送
- 心跳检测与连接保活
- 用户上下线状态通知

**核心技术点：**
- **连接池管理**：全局 Map 维护 userId → WebSocket Conn 映射
- **读写分离**：独立的 readLoop 和 writeLoop goroutine
- **优雅断线**：连接关闭时自动清理资源
- **背压控制**：256 容量的发送缓冲区，防止内存溢出

**连接管理架构：**
```go
// ws-gateway/internal/ws/conn.go
type Conn struct {
    UserID     int64
    WsConn     *websocket.Conn
    SendCh     chan []byte        // 发送缓冲区
    CloseCh    chan struct{}      // 关闭信号
    IsClosed   atomic.Bool
}

// ws-gateway/internal/ws/manager.go
type Manager struct {
    Connections sync.Map // userId -> *Conn
    mu          sync.RWMutex
}

// 添加连接
func (m *Manager) AddConn(userId int64, conn *Conn) {
    m.Connections.Store(userId, conn)
    go conn.readLoop()   // 接收客户端消息
    go conn.writeLoop()  // 发送服务器消息
}

// 推送消息
func (m *Manager) PushToUser(userId int64, message []byte) error {
    val, ok := m.Connections.Load(userId)
    if !ok {
        return errors.New("用户不在线")
    }
    conn := val.(*Conn)
    select {
    case conn.SendCh <- message:
        return nil
    case <-time.After(5 * time.Second):
        return errors.New("发送超时")
    }
}
```

**Kafka 消费者集成：**
```go
// ws-gateway/internal/kafka/consumer.go
func StartChatConsumer(manager *ws.Manager) {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers: []string{"localhost:9092"},
        Topic:   "im-chat-topic",
        GroupID: "ws-gateway-group",

    for {
        msg, err := reader.ReadMessage(context.Background())
        if err != nil {
            log.Printf("读取消息失败: %v", err)
            continue

        // 解析消息
        var chatMsg ChatMessage
        json.Unmarshal(msg.Value, &chatMsg)

            log.Printf("推送失败(userId=%d): %v", chatMsg.ToUserId, err)
            // TODO: 存储为离线消息
        }
    }
}
```

## 三、数据流转设计

### 3.1 消息发送完整链路

```
1. 客户端 A 发送消息
   ↓
2. WS Gateway 接收 WebSocket 消息
   ↓
3. WS Gateway 调用 Chat Service gRPC (SendMessage)
   ↓
4. Chat Service 检查拉黑关系
   ↓
5. Chat Service 生成消息 ID、时间戳
   ↓
6. Chat Service 投递消息到 Kafka (topic: im-chat-topic, key: recipient_id)
   ↓
7. WS Gateway Kafka Consumer 消费消息
   ↓
8. 查找接收者 B 的 WebSocket 连接
   ↓
9. 通过 SendCh 推送到 B 的 writeLoop
   ↓
10. B 客户端接收到实时消息
```

**关键优化点：**
- ✅ Kafka 分区键为 recipient_id，保证单个用户的消息顺序性
- ✅ Kafka 持久化保证消息不丢失（副本因子 = 3）
- ✅ WebSocket 异步发送，避免阻塞其他消息

### 3.2 好友请求处理流程

```
1. 用户 A 发起好友请求 (AddFriend)
   ↓
2. Friend Service 插入 friend_requests 表
   ↓
3. 发送通知到 Kafka (topic: im-friend-topic)
   ↓
4. WS Gateway 消费事件
   ↓
5. 推送实时通知给用户 B
   ↓
6. B 响应请求 (RespondFriendRequest: accept/reject)
   ↓
7. Friend Service 更新请求状态
   ↓
8. 如果同意：插入双向好友关系到 friends 表
   ↓
9. 通过 Kafka 通知 A 请求已被接受
```

### 3.3 用户登录状态管理

```
1. 用户登录成功 → Auth Service 返回 JWT Token
   ↓
2. 客户端携带 Token 建立 WebSocket 连接
   ↓
3. WS Gateway 验证 Token（调用 Auth Service VerifyToken）
   ↓
4. 验证通过后创建 Conn 对象，加入连接池
   ↓
5. Redis 设置在线状态：SET user:online:{uid} 1 EX 300
   ↓
6. 启动心跳机制：每 30 秒 PING-PONG，刷新 Redis TTL
   ↓
7. 用户断线时：
   - 触发 Conn.Close()
   - 从连接池移除
   - Redis 删除在线状态：DEL user:online:{uid}
```

## 五、高可用与性能优化

### 5.1 服务高可用

**策略一：无状态服务设计**
- 所有微服务不保存本地状态
- 状态数据存储在 Redis/MySQL
- 支持任意节点故障自动切换

**策略二：服务注册与发现**
- 使用 Etcd 作为注册中心
- 服务启动时自动注册
- 客户端通过 Etcd 动态发现可用节点

**策略三：熔断与限流**
- go-zero 内置熔断器
- 当下游服务故障时快速失败
- 限流保护核心服务不被打垮

### 5.2 缓存策略

**多级缓存架构：**
```
1. 本地缓存（进程内）：热点用户信息，TTL=30s
   ↓
2. Redis 缓存：用户信息、会话、在线状态，TTL=5min
   ↓
3. MySQL 持久化：最终数据源
```

**缓存更新策略：**
- **写操作**：先写 MySQL，再删除 Redis 缓存（Cache-Aside）
- **读操作**：先查 Redis，未命中则查 MySQL 并回写 Redis

### 5.3 消息队列优化

**Kafka 配置优化：**
```yaml
# 生产者配置
acks: all                    # 所有副本确认后才返回
compression: snappy          # 压缩以减少网络传输
batch.size: 16384            # 批量发送，提高吞吐量

# 消费者配置
group.id: ws-gateway-group   # 消费组支持水平扩展
enable.auto.commit: false    # 手动提交，确保至少一次消费
max.poll.records: 500        # 每次拉取消息数
```

**消息顺序性保证：**
- 使用 recipient_id 作为分区键
- 同一用户的所有消息路由到同一分区
- 分区内消息严格有序

### 5.4 数据库优化

**读写分离：**
- 主库处理写操作
- 从库处理读操作（用户信息查询、历史消息查询）
- 使用 GORM 的 `Replicas` 配置实现

**分库分表策略（预留）：**
- messages 表按用户 ID 哈希分表（16 张表）
- 单表最多存储 1000 万条消息
- 查询时通过中间件自动路由到对应分片

## 六、安全性设计

### 6.1 认证安全

- **密码存储**：bcrypt + salt（成本因子 12）
- **Token 安全**：JWT 签名，HS256 算法
- **传输加密**：WebSocket 支持 WSS (TLS)
- **防重放攻击**：Token 包含过期时间 + 设备指纹

### 6.2 接口防护

- **请求频率限制**：单用户 100 req/min
- **防刷机制**：同一账号 1 分钟内最多登录 5 次
- **输入校验**：Protobuf 自动校验 + 业务逻辑二次验证

### 6.3 数据隔离

- **租户隔离**：每个用户只能访问自己的数据
- **权限校验**：gRPC 拦截器统一鉴权
- **SQL 注入防护**：使用参数化查询（GORM 自动处理）
