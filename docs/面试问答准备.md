# IM即时通讯系统 - 面试问答准备

## 一、项目介绍（2-3分钟开场白）

### 标准回答模板

**面试官**：请介绍一下你的 IM 即时通讯系统项目。

**回答**：

这是我独立开发的一个**基于 Go 语言的微服务架构 IM 即时通讯系统**，项目完整实现了用户认证、好友管理、实时聊天等核心功能。

**技术架构方面**，我采用了 go-zero 微服务框架，将系统拆分为 5 个独立服务：
- **Auth Service** 负责用户认证和 JWT Token 管理
- **User Service** 负责用户信息管理
- **Chat Service** 负责消息的处理和分发
- **Friend Service** 负责好友关系和黑名单管理
- **WebSocket Gateway** 负责维护客户端长连接和实时推送

**核心技术栈包括**：
- **gRPC + Protobuf** 用于微服务间通信，比 HTTP+JSON 性能提升 3-5 倍
- **Kafka** 作为消息队列，实现消息的异步分发和削峰填谷
- **WebSocket** 实现客户端和服务器的双向实时通信
- **Redis** 用于缓存用户会话、在线状态等热点数据
- **MySQL** 持久化存储用户、消息、好友关系等数据

**架构亮点**有三个：
1. **消息可靠性**：通过 Kafka 持久化保证消息不丢失，副本因子设置为 3
2. **消息顺序性**：使用接收者 ID 作为 Kafka 分区键，保证单用户消息严格有序
3. **高并发支持**：WebSocket 连接池 + 读写分离设计，单机可支持 10 万并发连接

**项目难点**主要在两个方面：
1. **WebSocket 连接管理**：需要处理心跳保活、优雅断线、连接池并发安全等问题，我使用了 Go 的 sync.Map 和 goroutine 来实现高效的连接管理
2. **消息实时推送**：消息从发送到接收的整个链路涉及 WebSocket Gateway、gRPC、Kafka、数据库多个组件，我通过合理的架构设计将延迟控制在 100ms 以内

**业务价值**：这个项目让我深入理解了微服务架构、消息队列、长连接管理等后端核心技术，也具备了独立设计和实现复杂系统的能力。

## 二、技术深度问题

### 2.1 微服务架构

#### Q: 为什么要采用微服务架构？有什么优缺点？

**回答**：

**采用微服务的原因**：
1. **业务复杂度高**：IM 系统包含认证、消息、好友、群组等多个子系统，单体应用难以维护
2. **独立扩展需求**：聊天服务流量大，需要单独扩容；用户服务相对稳定，微服务可以针对性扩展
3. **技术演进灵活**：未来可以用其他语言重写某个服务，不影响整体

**优点**：
- **故障隔离**：某个服务挂了不影响其他服务，比如好友服务故障，聊天功能仍可用
- **独立部署**：可以单独升级某个服务，不需要整体重启
- **团队协作**：不同团队可以负责不同服务，降低协作成本

**缺点及应对**：
- **分布式事务复杂**：我采用了最终一致性方案，通过 Kafka 事件驱动来保证数据一致
- **网络开销**：服务间调用变多，我通过 gRPC 的高性能和合理的服务拆分来缓解
- **运维成本**：需要服务发现、监控、日志等基础设施，我用 Etcd + Prometheus 来解决

**实际案例**：
在开发过程中，我发现聊天服务压力大，就单独为它部署了 3 个实例做负载均衡，而认证服务只部署 1 个就够了，这就是微服务的灵活性。

#### Q: 服务之间如何通信？为什么选择 gRPC 而不是 HTTP？

**回答**：

**gRPC 的优势**：

| 维度 | gRPC + Protobuf | HTTP + JSON |
|------|----------------|-------------|
| **性能** | 二进制序列化，3-5 倍于 JSON | 文本格式，需要解析 |
| **类型安全** | 强类型定义，编译期检查 | 弱类型，运行时报错 |
| **网络开销** | 小（HTTP/2 多路复用） | 大（HTTP/1.1 短连接） |
| **代码生成** | 自动生成客户端/服务端代码 | 需要手写 |

**实际案例**：
```go
// gRPC 调用示例（WebSocket Gateway 调用 Auth Service）
func (h *Handler) verifyToken(token string) (int64, error) {
    ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
    defer cancel()
    
    resp, err := h.authClient.VerifyToken(ctx, &auth.VerifyTokenRequest{
        AccessToken: token,
    })
    if err != nil {
        return 0, err
    }
    return resp.UserId, nil
}
```

**为什么不用 HTTP**：
- HTTP REST API 更适合对外暴露给前端
- 内部服务通信用 gRPC 效率更高
- 我的架构中，前端调用 API Gateway（HTTP），Gateway 再通过 gRPC 调用后端服务

### 2.2 消息队列

#### Q: 为什么用 Kafka 而不是 Redis/RabbitMQ？

**回答**：

**三种 MQ 对比**：

| 特性 | Kafka | RabbitMQ | Redis |
|------|-------|----------|-------|
| **吞吐量** | 100 万/秒 | 10 万/秒 | 10 万/秒 |
| **持久化** | 磁盘，可靠 | 磁盘，可靠 | 内存，易丢失 |
| **消息顺序** | 分区内有序 | 队列有序 | 无保证 |
| **应用场景** | 大数据、日志、事件流 | 任务队列、RPC | 缓存、计数器 |

**选择 Kafka 的原因**：
1. **高吞吐量**：IM 系统消息量大，Kafka 单机可达 100 万 TPS
2. **持久化保证**：消息写入磁盘，即使消费者宕机也不丢失，可以回溯消费
3. **分区机制**：天然支持水平扩展，增加分区就能提高并发
4. **消息顺序性**：同一用户的消息路由到同一分区，保证严格有序

**实际使用**：
```go
// 生产者发送消息（Chat Service）
err := producer.WriteMessages(context.Background(),
    kafka.Message{
        Topic: "im-chat-topic",
        Key:   []byte(strconv.FormatInt(toUserId, 10)), // 分区键
        Value: msgBytes,
    },
)

// 消费者接收消息（WS Gateway）
reader := kafka.NewReader(kafka.ReaderConfig{
    Brokers: []string{"localhost:9092"},
    Topic:   "im-chat-topic",
    GroupID: "ws-gateway-group",  // 消费组支持多实例
})
```

---

#### Q: Kafka 如何保证消息不丢失？

**回答**：

**三层保障机制**：

1. **生产者层面**：
```go
producer := &kafka.Writer{
    Brokers: []string{"localhost:9092"},
    Topic:   "im-chat-topic",
    // 关键配置
    RequiredAcks: kafka.RequireAll,  // 等待所有副本确认
    Compression:  kafka.Snappy,      // 压缩以提高性能
}
```
- `RequiredAcks = -1`：所有副本都写入成功才返回成功
- 如果发送失败，返回错误，业务层可以重试

2. **Broker 层面**：
```yaml
# Kafka 配置
replication.factor: 3           # 副本因子 = 3，容忍 2 台机器故障
min.insync.replicas: 2          # 至少 2 个副本同步成功
log.retention.hours: 168        # 消息保留 7 天
```

3. **消费者层面**：
```go
// 手动提交 offset，处理成功后再提交
for {
    msg := reader.FetchMessage(context.Background())
    
    // 处理消息
    manager.PushToUser(userId, msg.Value)
    
    // 处理成功后才提交
    reader.CommitMessages(context.Background(), msg)
}
```

**极端场景处理**：
- 如果 Kafka 全挂了？→ 服务端会返回发送失败，客户端可以提示用户重试
- 如果消费者处理失败？→ 不提交 offset，下次消费会重新处理（至少一次消费）

#### Q: 如何保证消息的顺序性？

**回答**：

**核心原理**：Kafka 分区内消息严格有序，但不同分区之间无序

**我的实现**：
```go
// 以接收者 userId 作为分区键
key := []byte(strconv.FormatInt(toUserId, 10))

err := producer.WriteMessages(context.Background(),
    kafka.Message{
        Topic: "im-chat-topic",
        Key:   key,  // 相同 key 路由到同一分区
        Value: msgBytes,
    },
)
```

**效果**：
- 用户 A 给用户 B 发 3 条消息："你好" → "在吗" → "约吗"
- 因为 toUserId = B，3 条消息都路由到同一分区（比如分区 5）
- 分区 5 内消息严格按时间顺序：1 → 2 → 3
- 消费者消费分区 5 时，按顺序处理，保证 B 收到的消息顺序正确

**为什么不用全局顺序**？
- 全局顺序需要单分区，吞吐量太低（只有几千 TPS）
- IM 系统只需要保证单个用户的消息有序，不需要全局有序
- 我用 16 个分区，吞吐量可达百万级

**图示**：
```
消息流：
A → B: "你好"   (hash(B) % 16 = 5) → 分区 5
A → B: "在吗"   (hash(B) % 16 = 5) → 分区 5
A → B: "约吗"   (hash(B) % 16 = 5) → 分区 5

分区 5：["你好", "在吗", "约吗"] ← 严格有序
```

### 2.3 WebSocket 长连接

#### Q: 为什么用 WebSocket 而不是 HTTP 轮询？

**回答**：

**对比表**：

| 方式 | 延迟 | 服务器压力 | 带宽消耗 | 实现复杂度 |
|------|------|-----------|---------|----------|
| **HTTP 轮询** | 1-5 秒 | 极高（每秒大量请求） | 高 | 低 |
| **HTTP 长轮询** | 100-500ms | 高 | 中 | 中 |
| **WebSocket** | 10-100ms | 低（一次连接） | 低 | 高 |

**WebSocket 优势**：
1. **低延迟**：消息到达立即推送，不需要等待客户端轮询
2. **低开销**：建立连接后只传输数据帧，没有 HTTP 头部开销
3. **双向通信**：服务器可以主动推送，不需要客户端请求

**实际数据对比**：
```
HTTP 短轮询（每秒请求 1 次）：
- 带宽：每次请求 ~500 字节（HTTP 头）× 1 次/秒 = 500 字节/秒
- 10 万用户 = 50 MB/秒 = 4.3 TB/天

WebSocket：
- 建立连接：一次性开销（HTTP 头 + 握手）
- 数据传输：只有消息内容，无额外头部
- 10 万用户心跳（30 秒/次）= 0.3 MB/秒 = 26 GB/天

节省带宽：99%
```

#### Q: WebSocket 连接如何管理？如何处理断线重连？

**回答**：

**连接池设计**：
```go
type Manager struct {
    Connections sync.Map  // map[userId]*Conn，线程安全
}

type Conn struct {
    UserID   int64
    WsConn   *websocket.Conn
    SendCh   chan []byte  // 发送缓冲区（256 容量）
    CloseCh  chan struct{}
    IsClosed atomic.Bool
}
```

**关键点**：
1. **sync.Map**：Go 官方并发安全 Map，读多写少场景性能优于传统锁
2. **缓冲通道**：256 容量的 SendCh，防止突发消息阻塞
3. **原子操作**：atomic.Bool 避免重复关闭连接

**生命周期管理**：
```go
// 1. 建立连接
func (h *Handler) HandleWS(w http.ResponseWriter, r *http.Request) {
    // 验证 Token
    userId, err := h.verifyToken(r.Header.Get("Authorization"))
    
    // 升级为 WebSocket
    wsConn, _ := upgrader.Upgrade(w, r, nil)
    
    // 创建连接对象
    conn := NewConn(userId, wsConn)
    h.manager.AddConn(userId, conn)
    
    // 启动读写循环
    go conn.readLoop()
    go conn.writeLoop()
}

// 2. 心跳保活
func (c *Conn) writeLoop() {
    ticker := time.NewTicker(30 * time.Second)
    for {
        select {
        case msg := <-c.SendCh:
            c.WsConn.WriteMessage(websocket.TextMessage, msg)
        case <-ticker.C:
            c.WsConn.WriteMessage(websocket.PingMessage, nil)  // 心跳
        case <-c.CloseCh:
            return
        }
    }
}

// 3. 优雅关闭
func (c *Conn) Close() {
    if c.IsClosed.Swap(true) {
        return  // 已关闭，避免重复
    }
    close(c.CloseCh)
    c.WsConn.Close()
    // Manager 自动移除连接
}
```

**断线重连策略**（客户端）：
```javascript
class IMClient {
    connect() {
        this.ws = new WebSocket('ws://localhost:8080/ws');
        
        this.ws.onclose = () => {
            // 指数退避重连
            const delay = Math.min(1000 * (2 ** this.retryCount), 30000);
            setTimeout(() => this.connect(), delay);
            this.retryCount++;
        };
        
        this.ws.onopen = () => {
            this.retryCount = 0;  // 重置重试次数
        };
    }
}
```

#### Q: 如何实现 WebSocket 的水平扩展？

**回答**：

**挑战**：多台 WebSocket 服务器，用户连接在机器 A，消息到达机器 B，怎么推送？

**解决方案**：Redis Pub/Sub

```
┌─────────────┐
│ WS Server A │  用户 123 连接在这台
└──────┬──────┘
       │ 订阅 Redis 频道
       ▼
┌─────────────────┐
│  Redis Pub/Sub  │
│  Channel: ws    │
└──────┬──────────┘
       │ 发布消息
       ▼
┌─────────────┐
│ WS Server B │  Kafka 消费到消息，发布到 Redis
└─────────────┘
```

**实现代码**：
```go
// WS Server B 从 Kafka 消费到消息
func handleKafkaMessage(msg kafka.Message) {
    var chatMsg ChatMessage
    json.Unmarshal(msg.Value, &chatMsg)
    
    // 先尝试本地推送
    if err := localManager.PushToUser(chatMsg.ToUserId, msg.Value); err != nil {
        // 本地没有连接，发布到 Redis
        redis.Publish(context.Background(), "ws:push", msg.Value)
    }
}

// 所有 WS Server 订阅 Redis 频道
func subscribeRedis() {
    pubsub := redis.Subscribe(context.Background(), "ws:push")
    for msg := range pubsub.Channel() {
        var chatMsg ChatMessage
        json.Unmarshal([]byte(msg.Payload), &chatMsg)
        
        // 如果用户连接在本机，推送
        localManager.PushToUser(chatMsg.ToUserId, []byte(msg.Payload))
    }
}
```

**优化方案**：一致性哈希
```go
// 计算用户应该连接到哪台服务器
serverIndex := hash(userId) % serverCount

// 客户端连接时，路由到对应服务器
// Nginx 配置
upstream ws_servers {
    hash $arg_userId consistent;
    server ws1.example.com;
    server ws2.example.com;
    server ws3.example.com;
}
```

**好处**：大部分消息直接推送，只有少量需要通过 Redis 转发

### 2.4 数据库设计

#### Q: 聊天记录表如何设计？如何优化查询性能？

**回答**：

**表结构**：
```sql
CREATE TABLE messages (
    id VARCHAR(36) PRIMARY KEY COMMENT 'UUID',
    from_user_id BIGINT NOT NULL,
    to_user_id BIGINT NOT NULL,
    content TEXT NOT NULL,
    msg_type TINYINT DEFAULT 1 COMMENT '1=文本 2=图片 3=文件',
    status TINYINT DEFAULT 0 COMMENT '0=未读 1=已读',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- 索引
    INDEX idx_conversation (from_user_id, to_user_id, created_at),
    INDEX idx_to_user (to_user_id, status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**核心查询优化**：

**场景 1：查询两人聊天记录**
```sql
-- 查询 A 和 B 的最近 20 条消息
SELECT * FROM messages 
WHERE (from_user_id = 123 AND to_user_id = 456)
   OR (from_user_id = 456 AND to_user_id = 123)
ORDER BY created_at DESC 
LIMIT 20;

-- 使用 idx_conversation 索引，只扫描相关行
```

**场景 2：查询未读消息数**
```sql
-- 查询用户 123 的未读消息数
SELECT COUNT(*) FROM messages 
WHERE to_user_id = 123 AND status = 0;

-- 使用 idx_to_user 索引，只扫描未读消息
```

**分页优化**：
```sql
-- 传统分页（效率低）
SELECT * FROM messages WHERE to_user_id = 123 
ORDER BY created_at DESC LIMIT 20 OFFSET 2000;
-- 问题：扫描 2020 行，丢弃 2000 行

-- 游标分页（高效）
SELECT * FROM messages WHERE to_user_id = 123 
AND created_at < '2026-02-03 10:00:00'  -- 上一页最后一条的时间
ORDER BY created_at DESC LIMIT 20;
-- 优势：始终只扫描 20 行
```

**数据量大时的分表策略**：
```go
// 按用户 ID 哈希分表（16 张表）
tableIndex := userId % 16
tableName := fmt.Sprintf("messages_%02d", tableIndex)

// 查询时路由到对应分表
db.Table(tableName).Where("to_user_id = ?", userId).Find(&messages)
```

#### Q: 如何设计好友关系表？为什么要双向存储？

**回答**：

**表结构**：
```sql
CREATE TABLE friends (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    friend_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_friend (user_id, friend_id),
    INDEX idx_user (user_id)
) ENGINE=InnoDB;
```

**为什么双向存储**？

**场景对比**：

方案一：单向存储（user_id < friend_id）
```sql
-- A 添加 B：只插入一条 (123, 456)
INSERT INTO friends (user_id, friend_id) VALUES (123, 456);

-- 查询 A 的好友：需要 UNION
SELECT friend_id FROM friends WHERE user_id = 123
UNION
SELECT user_id FROM friends WHERE friend_id = 123;
-- 问题：两次查询，效率低
```

方案二：双向存储（我的方案）
```sql
-- A 添加 B：插入两条记录
INSERT INTO friends (user_id, friend_id) VALUES (123, 456), (456, 123);

-- 查询 A 的好友：一次查询
SELECT friend_id FROM friends WHERE user_id = 123;
-- 优势：直接命中索引 idx_user
```

**代码实现**：
```go
func (l *AddFriendLogic) AddFriend(req *friend.AddFriendRequest) error {
    // 开启事务
    tx := l.svcCtx.DB.Begin()
    
    // 插入双向关系
    tx.Create(&Friend{UserId: req.UserId, FriendId: req.FriendId})
    tx.Create(&Friend{UserId: req.FriendId, FriendId: req.UserId})
    
    // 提交事务
    return tx.Commit().Error
}
```

**trade-off**：
- 优点：查询快（单次查询）
- 缺点：存储空间翻倍
- 结论：空间换时间，IM 系统查询频繁，值得

### 2.5 认证与安全

#### Q: JWT Token 如何设计？为什么用双 Token 机制？

**回答**：

**双 Token 架构**：

| Token 类型 | 有效期 | 存储位置 | 用途 | 可撤销性 |
|-----------|-------|---------|------|---------|
| **AccessToken** | 15 分钟 | 内存/LocalStorage | API 调用 | 否（有效期内始终有效） |
| **RefreshToken** | 7 天 | HttpOnly Cookie | 刷新 AccessToken | 是（存在 Redis/数据库） |

**设计理由**：

**问题 1**：如果只用一个长期 Token（7 天）？
- 风险：Token 泄漏后，攻击者可以使用 7 天
- 无法主动撤销（除非加入黑名单，查询开销大）

**问题 2**：如果只用一个短期 Token（15 分钟）？
- 用户体验差：每 15 分钟就要重新登录

**双 Token 方案**：
- AccessToken 短期（15 分钟）：即使泄漏，影响有限
- RefreshToken 长期（7 天）：用户体验好，且可撤销（存在服务端）

**实现流程**：
```
1. 用户登录
   ↓
2. 生成 AccessToken + RefreshToken
   ↓
3. RefreshToken 存入 Redis（key: refresh:{userId}:{deviceId}）
   ↓
4. 返回两个 Token 给客户端
   ↓
5. 客户端每次请求携带 AccessToken
   ↓
6. AccessToken 过期（15 分钟后）
   ↓
7. 客户端用 RefreshToken 换取新的 AccessToken
   ↓
8. 服务端验证 RefreshToken（查 Redis）
   ↓
9. 生成新的 AccessToken，RefreshToken 不变
```

**代码示例**：
```go
// 生成 Token
func GenerateTokens(userId int64, deviceId string) (string, string, error) {
    // Access Token
    accessClaims := jwt.MapClaims{
        "user_id":   userId,
        "device_id": deviceId,
        "exp":       time.Now().Add(15 * time.Minute).Unix(),
    }
    accessToken, _ := jwt.NewWithClaims(jwt.SigningMethodHS256, accessClaims).
        SignedString([]byte(jwtSecret))
    
    // Refresh Token
    refreshClaims := jwt.MapClaims{
        "user_id": userId,
        "exp":     time.Now().Add(7 * 24 * time.Hour).Unix(),
    }
    refreshToken, _ := jwt.NewWithClaims(jwt.SigningMethodHS256, refreshClaims).
        SignedString([]byte(jwtSecret))
    
    // 存入 Redis
    redis.Set(fmt.Sprintf("refresh:%d:%s", userId, deviceId), refreshToken, 7*24*time.Hour)
    
    return accessToken, refreshToken, nil
}

// 刷新 Token
func RefreshAccessToken(refreshToken string) (string, error) {
    // 解析 Refresh Token
    claims, err := jwt.Parse(refreshToken)
    userId := claims["user_id"].(int64)
    
    // 验证 Redis 中是否存在
    storedToken, _ := redis.Get(fmt.Sprintf("refresh:%d:*", userId)).Result()
    if storedToken != refreshToken {
        return "", errors.New("无效的 Refresh Token")
    }
    
    // 生成新的 Access Token
    return GenerateAccessToken(userId, deviceId)
}
```

**强制下线**：
```go
// 删除 Redis 中的 Refresh Token
redis.Del(fmt.Sprintf("refresh:%d:%s", userId, deviceId))

// 用户下次刷新时会失败，需要重新登录
```

#### Q: 如何防止密码泄漏？bcrypt 原理是什么？

**回答**：

**错误做法**：
```go
// 明文存储（绝对不行）
password := "123456"
db.Create(&User{Password: password})

// MD5/SHA256（不安全）
password := md5.Sum([]byte("123456"))
// 问题：彩虹表攻击，秒破
```

**正确做法：bcrypt**
```go
import "golang.org/x/crypto/bcrypt"

// 注册时加密
func Register(username, password string) error {
    hashedPassword, _ := bcrypt.GenerateFromPassword(
        []byte(password), 
        12,  // cost，越大越安全但越慢
    )
    db.Create(&User{
        Username: username,
        Password: string(hashedPassword),
    })
}

// 登录时验证
func Login(username, password string) bool {
    user := getUserByUsername(username)
    err := bcrypt.CompareHashAndPassword(
        []byte(user.Password), 
        []byte(password),
    )
    return err == nil
}
```

**bcrypt 三大特性**：

1. **加盐（Salt）**：
   - 每次生成的哈希都不同，即使密码相同
   - 例如："123456" → "$2a$12$abc..." 和 "$2a$12$xyz..." 完全不同
   - 防止彩虹表攻击

2. **自适应（Adaptive）**：
   - cost 参数控制计算复杂度
   - cost=12 意味着 2^12 = 4096 次迭代
   - 硬件升级后可以提高 cost

3. **慢哈希（Slow Hash）**：
   - 单次计算耗时 ~100ms
   - 正常登录无感知（100ms 可接受）
   - 暴力破解成本极高（1000 次/秒 → 10 次/秒）

**实际效果**：
```
密码：123456

MD5：
- 第 1 次：e10adc3949ba59abbe56e057f20f883e
- 第 2 次：e10adc3949ba59abbe56e057f20f883e（相同）
- 彩虹表一查就破

bcrypt：
- 第 1 次：$2a$12$R9h/cIPz0gi.URNNX3kh2OPST9/PgBkqquzi.Ss7KIUgO2t0jWMUW
- 第 2 次：$2a$12$2AFBsN1q.E0I7wXe5dJp2O1/qJqQz9yCJt7BhzYo3Z5Rb1zrRqNPe（不同）
- 无法用彩虹表破解
```

## 三、项目难点与解决方案

### 3.1 难点一：消息实时性与可靠性的平衡

**问题描述**：
- 实时性要求：消息要尽快送达，延迟 < 100ms
- 可靠性要求：消息不能丢失，即使用户离线也要能收到

**矛盾点**：
- 直接 WebSocket 推送：快但不可靠（用户离线就丢失）
- 先存数据库再推送：可靠但慢（数据库写入耗时）

**我的解决方案**：
```
1. 消息先发到 Kafka（异步，5ms）
   ↓
2. Kafka Consumer 消费消息（实时，10ms）
   ↓
3. 推送到 WebSocket（实时，5ms）
   ↓
4. 异步写入 MySQL（后台任务，不阻塞）
   ↓
5. 如果用户离线，存入 Redis 离线队列
```

**核心代码**：
```go
// 1. 快速返回（不等数据库）
func SendMessage(req *SendMessageRequest) (*SendMessageResponse, error) {
    // 发到 Kafka 就返回
    kafka.Produce(topic, msg)
    return &SendMessageResponse{MessageId: msg.Id, Success: true}, nil
}

// 2. 异步持久化
go func() {
    for msg := range kafkaConsumer {
        // 后台写入数据库
        db.Create(&Message{...})
    }
}()

// 3. 离线消息处理
if err := wsManager.PushToUser(userId, msg); err != nil {
    redis.RPush(fmt.Sprintf("offline:%d", userId), msg)
}
```

**效果**：
- 在线用户：20-50ms 收到消息
- 离线用户：上线后推送离线消息
- 消息不丢失：Kafka 持久化 + MySQL 备份

### 3.2 难点二：WebSocket 连接池的并发安全

**问题描述**：
多个 goroutine 同时操作连接池（添加、删除、查找），如何保证线程安全？

**错误方案**：
```go
type Manager struct {
    connections map[int64]*Conn
    mu          sync.RWMutex
}

// 问题：锁粒度大，高并发下性能差
func (m *Manager) AddConn(userId int64, conn *Conn) {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.connections[userId] = conn
}

func (m *Manager) GetConn(userId int64) (*Conn, bool) {
    m.mu.RLock()
    defer m.mu.RUnlock()
    conn, ok := m.connections[userId]
    return conn, ok
}
```

**我的优化方案**：
```go
type Manager struct {
    connections sync.Map  // Go 官方并发安全 Map
}

func (m *Manager) AddConn(userId int64, conn *Conn) {
    m.connections.Store(userId, conn)  // 无锁操作
}

func (m *Manager) GetConn(userId int64) (*Conn, bool) {
    val, ok := m.connections.Load(userId)
    if !ok {
        return nil, false
    }
    return val.(*Conn), true
}

func (m *Manager) PushToUser(userId int64, msg []byte) error {
    conn, ok := m.GetConn(userId)
    if !ok {
        return errors.New("用户不在线")
    }
    
    // 非阻塞发送
    select {
    case conn.SendCh <- msg:
        return nil
    case <-time.After(5 * time.Second):
        return errors.New("发送超时")
    }
}
```

**sync.Map 优势**：
- 读多写少场景性能优于传统锁
- IM 系统正好符合：频繁查找连接（读），偶尔添加/删除（写）
- 测试结果：QPS 提升 3 倍

### 3.3 难点三：Kafka 消费延迟问题

**问题描述**：
高峰期消息积压，Kafka Consumer Lag 达到 10 万条，用户消息延迟 10 秒以上

**排查过程**：
```bash
# 查看 Consumer Lag
kafka-consumer-groups.sh --describe --group ws-gateway-group

# 结果
TOPIC           PARTITION  LAG
im-chat-topic   0          25000
im-chat-topic   1          30000
im-chat-topic   2          28000
...
```

**根因分析**：
1. 消费者实例只有 1 个，消费能力不足
2. 每条消息处理耗时 50ms（推送 WebSocket + 数据库写入）
3. 生产速度 1000 msg/s，消费速度只有 20 msg/s

**我的解决方案**：

**优化 1：增加消费者实例**
```bash
# 原来：1 个实例
ws-gateway-1

# 优化后：3 个实例（Docker Compose）
docker-compose scale ws-gateway=3

# 效果：Lag 从 10 万降到 3 万
```

**优化 2：批量消费**
```go
// 原来：逐条处理
for {
    msg := reader.ReadMessage()
    processMessage(msg)  // 50ms
}

// 优化后：批量处理
for {
    msgs := reader.FetchBatch(context.Background(), 100, time.Second)
    
    // 并发处理
    var wg sync.WaitGroup
    for _, msg := range msgs {
        wg.Add(1)
        go func(m kafka.Message) {
            defer wg.Done()
            processMessage(m)
        }(msg)
    }
    wg.Wait()
    
    reader.CommitMessages(context.Background(), msgs...)
}
```

**优化 3：异步写数据库**
```go
// 原来：同步写（阻塞 30ms）
func processMessage(msg kafka.Message) {
    wsManager.Push(userId, msg.Value)  // 10ms
    db.Create(&Message{...})           // 30ms ← 瓶颈
}

// 优化后：异步写
func processMessage(msg kafka.Message) {
    wsManager.Push(userId, msg.Value)  // 10ms
    
    // 异步写数据库
    go func() {
        db.Create(&Message{...})
    }()
    // 不等待数据库，直接返回
}
```

**最终效果**：
- Lag 从 10 万降到 0
- 消息延迟从 10 秒降到 50ms
- 消费速度从 20 msg/s 提升到 2000 msg/s
