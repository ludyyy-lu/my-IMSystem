## 1. 功能概述

本文档描述 IM 系统「未读/已读」功能的完整实现方案，涵盖：

| 功能 | 说明 |
|------|------|
| **发送消息初始为未读** | 所有入库消息默认 `status=0`（未读） |
| **单条消息已读（ack）** | 用户通过 WebSocket 对指定消息 ID 发送 ack，服务端标记已读并推送已读回执给发送方 |
| **会话批量已读（batch_ack）** | 用户打开会话窗口时，一次性清零与某对端的全部未读消息 |
| **未读数统计查询** | 通过 gRPC 查询用户的未读消息总数，按会话（对端用户）聚合，含最新消息预览 |
| **已读回执实时推送** | 消息标记为已读后，通过 Kafka 异步通知 ws-gateway，将已读回执实时推送至消息发送方的 WebSocket 客户端 |
| **离线已读保障** | 若发送方当前离线，回执事件存入 Redis 离线队列，待其上线时立即下发 |

## 2. 架构设计

```
Client A (消息接收方)                  Client B (消息发送方)
     │                                       │
     │  WS: {"type":"ack","message_id":"X"}  │
     │                                       │
     ▼                                       │
 ws-gateway                                  │
  router.handleAck()                         │
     │                                       │
     │──gRPC──► chat-service                 │
               AckMessage()                  │
                  │                          │
                  ├─► DB: UPDATE message     │
                  │    SET status=1          │
                  │    WHERE id=X            │
                  │                          │
                  └─► Kafka: read-receipt    │
                       topic (ReadReceiptEvent)
                                  │
                           ws-gateway        │
                        (consume goroutine)  │
                                  │          │
                                  └──WS push─►
                           PushTypeReadReceipt
                              到 Client B
```

**组件职责：**

| 组件 | 职责 |
|------|------|
| `ws-gateway/router` | 接收客户端 WS 消息，路由到 ack / batch_ack handler |
| `chat-service/logic/AckMessageLogic` | 标记单条消息已读，发布 `ReadReceiptEvent` 到 Kafka |
| `chat-service/logic/BatchAckMessagesLogic` | 批量标记会话消息已读，发布 `BatchReadReceiptEvent` |
| `chat-service/logic/GetUnreadCountLogic` | 按会话聚合查询用户未读数 |
| `ws-gateway/consume` | 消费 read-receipt Kafka topic，将回执推送至发送方 |
| `ws-gateway/push.Service` | 若用户在线直接 WS 推送，否则存入 Redis 离线队列 |

## 3. 数据模型

### 3.1 消息表（MySQL）

```sql
CREATE TABLE message (
    id          BIGINT        PRIMARY KEY AUTO_INCREMENT,
    from_user_id BIGINT       NOT NULL COMMENT '发送方 ID',
    to_user_id   BIGINT       NOT NULL COMMENT '接收方 ID',
    content      TEXT         NOT NULL COMMENT '消息内容',
    msg_type     TINYINT      DEFAULT 1 COMMENT '消息类型（1=文本）',
    status       TINYINT      DEFAULT 0 COMMENT '0=未读 1=已读',
    created_at   DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    INDEX idx_from_to   (from_user_id, to_user_id),
    INDEX idx_to_status (to_user_id, status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**索引设计说明：**

- `idx_to_status`：加速「查询 userId 所有未读消息」场景（`WHERE to_user_id=? AND status=0`）。
- `idx_from_to`：加速「查询两个用户之间的历史消息」场景。

### 3.2 Kafka 消息格式

#### ReadReceiptEvent（单条已读回执）

```json
{
  "message_id": "123",
  "sender_id": 1001,
  "reader_id": 1002,
  "read_at": 1712345678000
}
```

#### BatchReadReceiptEvent（批量已读回执）

```json
{
  "sender_id": 1001,
  "reader_id": 1002,
  "acked_count": 5,
  "read_at": 1712345678000
}
```

### 3.3 proto 消息类型

```protobuf
message AckMessageReq   { string message_id = 1; int64 user_id = 2; }
message AckMessageResp  { string status = 1; }

message BatchAckReq     { int64 user_id = 1; int64 peer_id = 2; }
message BatchAckResp    { string status = 1; int64 acked_count = 2; }

message GetUnreadCountReq  { int64 user_id = 1; }
message UnreadConversation {
  int64 peer_id = 1; int64 unread_count = 2;
  string last_content = 3; int64 last_timestamp = 4;
}
message GetUnreadCountResp {
  int64 total_unread = 1;
  repeated UnreadConversation conversations = 2;
}
```

## 4. 核心流程

### 4.1 发送消息（初始未读状态）

```
Client A ──WS chat──► ws-gateway ──Kafka──► chat-service consumer
                                                 │
                                          db.Create(&Message{
                                            Status: 0,  // 未读
                                          })
```

所有新消息入库时 `status` 默认为 `0`（未读），无需额外操作。

---

### 4.2 单条消息已读（ack）

**客户端发送：**

```json
{"type": "ack", "message_id": "123"}
```

**服务端处理链路：**

```
ws-gateway/router.handleAck()
  → ChatRpc.AckMessage(messageId="123", userId=currentUser)
      → model.MarkMessageAsReadAndGet(db, "123", userId)
          → DB: SELECT FROM message WHERE id=123 AND to_user_id=userId
          → if status==1: return nil (幂等，不更新)
          → UPDATE message SET status=1 WHERE id=123
          → return &Message{FromUserId: senderId, ...}
      → if ReadReceiptProducer != nil && msg != nil:
          Kafka.SendMessage("read-receipt-topic", ReadReceiptEvent{
            MessageId: "123", SenderId: msg.FromUserId,
            ReaderId: userId, ReadAt: now,
          })
  → PushService.PushToUser(userId, "ack_result", {message_id, success})
```

**客户端收到（执行 ack 的用户）：**

```json
{"type": "ack_result", "payload": {"message_id": "123", "success": true}}
```

**消息发送方收到（通过 Kafka → ws-gateway 推送）：**

```json
{"type": "read_receipt", "payload": {"message_id": "123", "sender_id": 1001, "reader_id": 1002, "read_at": 1712345678000}}
```

---

### 4.3 会话批量已读（batch_ack）

适用场景：用户打开与某好友的聊天窗口，一次性清零所有未读。

**客户端发送：**

```json
{"type": "batch_ack", "peer_id": 1001}
```

**服务端处理：**

```
ws-gateway/router.handleBatchAck()
  → ChatRpc.BatchAckMessages(userId=currentUser, peerId=1001)
      → model.BatchMarkConversationAsRead(db, userId, peerId)
          → UPDATE message SET status=1
            WHERE from_user_id=peerId AND to_user_id=userId AND status=0
          → return rowsAffected
      → if ackedCount > 0:
          Kafka.SendMessage("read-receipt-topic", BatchReadReceiptEvent{...})
  → PushService.PushToUser(userId, "batch_ack_result", {peer_id, success, acked_count})
```

**客户端收到（执行 batch_ack 的用户）：**

```json
{"type": "batch_ack_result", "payload": {"peer_id": 1001, "success": true, "acked_count": 5}}
```

---

### 4.4 已读回执推送给发送方

```
Kafka read-receipt topic
    │
    ▼
ws-gateway/consume (goroutine: ws-gateway-read-receipt-group)
    │
    ├─ json.Unmarshal → BatchReadReceiptEvent (acked_count>0)
    │     └─ PushToUser(sender_id, "read_receipt", batch)
    │
    └─ json.Unmarshal → ReadReceiptEvent
          └─ PushToUser(sender_id, "read_receipt", single)
```

若 `sender_id` 在线：直接通过 WebSocket 推送。  
若 `sender_id` 离线：存入 Redis `offline:msg:{sender_id}` 列表，待其上线时 `deliverOfflineMessages` 批量下发。

---

### 4.5 查询未读消息数

通过 gRPC 调用（适用于 HTTP API 或内部服务查询）：

```go
resp, err := chatRpc.GetUnreadCount(ctx, &chat.GetUnreadCountReq{UserId: 1002})
// resp.TotalUnread = 8
// resp.Conversations = [
//   {PeerId:1001, UnreadCount:5, LastContent:"在吗？", LastTimestamp:1712345600000},
//   {PeerId:1003, UnreadCount:3, LastContent:"收到了吗？", LastTimestamp:1712345400000},
// ]
```

**SQL 实现：**

```sql
SELECT
  from_user_id,
  COUNT(*) AS cnt,
  SUBSTRING_INDEX(GROUP_CONCAT(content ORDER BY created_at DESC SEPARATOR '\x00'), '\x00', 1) AS last_msg,
  MAX(created_at) AS last_time
FROM message
WHERE to_user_id = ? AND status = 0
GROUP BY from_user_id
ORDER BY last_time DESC;
```

## 5. gRPC 接口说明

### chat.Chat 服务

| RPC | 请求 | 响应 | 说明 |
|-----|------|------|------|
| `SendMessage` | `SendMessageReq` | `SendMessageResp` | 发送消息（原有） |
| `GetChatHistory` | `GetChatHistoryReq` | `GetChatHistoryResp` | 获取历史消息（原有） |
| `AckMessage` | `AckMessageReq` | `AckMessageResp` | 单条消息已读回执 |
| `BatchAckMessages` | `BatchAckReq` | `BatchAckResp` | 批量会话已读 |
| `GetUnreadCount` | `GetUnreadCountReq` | `GetUnreadCountResp` | 查询未读消息统计 |

#### AckMessage

- **幂等**：对同一消息重复 ack 不会报错，`status` 保持 `1`，不重复发布 Kafka 事件。
- **权限**：只允许消息接收方（`to_user_id`）执行 ack；发送方调用不会更新 DB。

#### BatchAckMessages

- **幂等**：若该会话已全部已读，`acked_count=0`，不发布 Kafka 事件。
- **原子性**：单次 `UPDATE ... WHERE status=0` 操作，DB 层面原子。

#### GetUnreadCount

- **实时性**：直接查询 MySQL，反映当前最新状态。
- **若 userId=0**：直接返回空响应，不查 DB。

## 7. Kafka Topics 说明

| Topic | 生产者 | 消费者 | 消费组 | 消息格式 |
|-------|--------|--------|--------|----------|
| `chat-messages` | ws-gateway | chat-service | `chat-service-group` | `ChatMessage` JSON |
| `read-receipt` | chat-service | ws-gateway | `ws-gateway-read-receipt-group` | `ReadReceiptEvent` / `BatchReadReceiptEvent` JSON |

**分区策略：**
- `chat-messages`：使用 `LeastBytes` 负载均衡（轮询近似）。
- `read-receipt`：使用消息 ID 或 `batch:{senderId}:{readerId}` 作为 key，确保同一会话的回执有序。

**消费保障：**
- `read-receipt` 消费组使用 `ReadMessage`（自动提交 offset），适合「最少一次」语义。
- 若对「精确一次」有要求，可在业务层加幂等检查（DB 状态判断）。

## 10. 错误处理与幂等性

| 场景 | 处理方式 |
|------|----------|
| AckMessage：消息不存在 | 返回 `{status: "OK"}`，不报错（防止重放攻击暴露消息 ID） |
| AckMessage：消息已是已读 | `MarkMessageAsReadAndGet` 返回 `(nil, nil)`，不更新 DB，不发布 Kafka 事件 |
| AckMessage：非接收方调用 | `WHERE to_user_id=userId` 条件不匹配，DB 不更新，返回 `{status: "OK"}` |
| BatchAckMessages：已全部已读 | `RowsAffected=0`，不发布 Kafka 事件，返回 `{status: "OK", acked_count: 0}` |
| Kafka 发布失败 | 仅记录错误日志，不影响 RPC 返回给客户端的结果（已读回执可容忍丢失重试） |
| ws-gateway：ChatRpc 为 nil | 安全跳过，记录 error 日志 |
| ws-gateway：PushService 为 nil | 安全跳过（nil 检查保护） |
| 用户离线时收到 read_receipt | `push.Service` 写入 Redis 离线队列，下次上线时下发 |

## 11. 性能考量

### 11.1 已读操作

- **单条 ack**：单次 `SELECT + UPDATE`，在 `idx_to_status` 索引下为点查，≤1ms。
- **批量 ack**：单次 `UPDATE ... WHERE from_user_id=? AND to_user_id=? AND status=0`，利用 `idx_to_status`，扫描范围可控。

### 11.2 未读数查询

`GetUnreadCountByPeer` 使用 `GROUP BY + COUNT + GROUP_CONCAT`，在数据量较大时可能较慢。

**优化建议（中高流量场景）：**

1. **Redis 计数器缓存**：在标记已读时同步更新 Redis HASH `unread:{userId}` 中对应 `peerId` 的计数，查询时优先读 Redis。
2. **Redis Sorted Set 会话列表**：以时间戳为 score 维护会话排序，避免每次聚合查询。
3. **异步刷新**：通过 Kafka 消费异步更新 Redis 计数，削峰。

> 当前实现直接查 MySQL，适合 **低到中等流量**（日活 < 10 万）的生产环境。

### 11.3 Kafka 延迟

已读回执通过 Kafka 异步传递，预期延迟 < 100ms（同机房）。对实时性要求极高的场景，可考虑 ws-gateway 直接通过 gRPC 向目标用户的 gateway 节点发送回执（需实现 gateway 间路由）。

## 12. 运维与监控

### 12.1 关键日志

| 日志关键字 | 含义 | 告警级别 |
|-----------|------|----------|
| `AckMessage: mark failed` | 标记已读 DB 操作失败 | ERROR |
| `AckMessage: publish read receipt failed` | 发布 Kafka 事件失败 | WARN |
| `BatchAckMessages: DB update failed` | 批量标记 DB 失败 | ERROR |
| `failed to parse read receipt event` | ws-gateway 消费回执消息格式错误 | WARN |
| `batch_ack failed` | ws-gateway router batch_ack RPC 失败 | ERROR |

### 12.2 监控指标（建议）

| 指标 | 类型 | 说明 |
|------|------|------|
| `chat.ack.success_total` | Counter | 成功已读操作数 |
| `chat.ack.fail_total` | Counter | 失败已读操作数 |
| `chat.batch_ack.acked_count` | Histogram | 批量已读的消息数分布 |
| `kafka.read_receipt.publish_fail_total` | Counter | 回执 Kafka 发布失败数 |
| `ws.push.read_receipt_total` | Counter | ws-gateway 推送回执次数 |

### 12.3 Kafka Consumer Lag 告警

监控消费组 `ws-gateway-read-receipt-group` 的 lag，建议阈值：lag > 1000 触发告警，lag > 5000 触发 PagerDuty。

## 14. 升级迁移说明

### 14.1 数据库迁移

`gorm.AutoMigrate` 在服务启动时自动执行，会添加新索引。已有数据无需手动迁移。

### 14.2 Kafka Topic 创建

在生产环境部署前，需预先创建 `read-receipt` topic：

```bash
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --replication-factor 3 \
  --partitions 12 \
  --topic read-receipt
```

**分区数建议**：与 `chat-messages` topic 保持一致或略少（回执流量通常低于消息流量）。

### 14.3 配置更新

在 `chat.yaml` 和 `ws.yaml` 中添加 `ReadReceiptTopic: "read-receipt"`。

若不添加（保持原有配置），功能**降级**为：
- AckMessage 仍然正常标记 DB 已读。
- 不发布 Kafka 回执事件，消息发送方**不会实时收到**已读回执。
- 其他功能不受影响。

### 14.4 滚动发布顺序

1. 先发布 **chat-service**（新增 RPC 方法，向下兼容旧客户端）。
2. 再发布 **ws-gateway**（开始消费 read-receipt topic，处理新 WS 消息类型）。
3. 客户端可以随后更新（新增 ack / batch_ack 发送逻辑）。
